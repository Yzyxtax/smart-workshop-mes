## 1. 架构概述

### 1.1 三层模型

```
┌──────────────────────────────────────────────────────────────┐
│                      Plan (生产计划)                           │
│  编号: PLAN-yyyyMMdd-nnn                                     │
│  粒度: 产品×时间段                                              │
│  状态驱动: 生产主管手动                                       │
│  向下输出: 按工艺流程+产线拆分为 Order                           │
│  向上聚合: 来自 Order 的状态同步                                │
├──────────────────────────────────────────────────────────────┤
│                      Order (生产订单)                          │
│  编号: {planNo}-{lineNo}                                     │
│  粒度: 计划×产线                                               │
│  状态驱动: 系统联动(发布) + 主管(干预) + 工单事实(执行)           │
│  向下输出: 按生产线工艺流程生成 WorkOrder                        │
│  向上聚合: 来自 WorkOrder 的状态同步                             │
├──────────────────────────────────────────────────────────────┤
│                   WorkOrder (工单)                             │
│  编号: {orderNo}-{processId}-{seq}                           │
│  粒度: 订单×工序×人员                                          │
│  状态驱动: 员工手动确认(开始/完成) + 主管(干预)                   │
│  向下: 最底层，无子对象                                         │
│  向上: 关键工单状态聚合 → Order                                 │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 核心级联链

```
Plan PUBLISH → Order CREATED → Order PUBLISH(系统) → WorkOrder CREATED → WorkOrder PUBLISH(系统)
                                                                              │
Plan RUNNING ← Order RUNNING ←──────────────────── WorkOrder START_WORK(员工)
Plan PAUSED  ← Order PAUSED  ←──────────────────── WorkOrder PAUSE
Plan COMPLETED ← Order COMPLETED ←───────────────── WorkOrder FINISH_WORK(员工)
```

### 1.3 层级对比表

| 维度 | Plan | Order | WorkOrder |
|------|------|-------|-----------|
| **定位** | 宏观计划调度 | 产线调度单元 | 现场执行单元 |
| **状态粒度** | 产品×时间 | 计划×产线 | 订单×工序×人员 |
| **PUBLISH 触发** | 人工(生产主管) | 系统联动 | 系统联动 |
| **START_WORK 触发** | ❌ | 工单事实驱动 | **员工手动确认** |
| **FINISH_WORK 触发** | ❌ | 工单事实驱动 | **员工手动确认** |
| **PAUSE/RESUME** | 生产主管 | 车间主任/生产主管 | 员工(userId匹配)或主管 |
| **TERMINATE** | 生产主管 | 车间主任/生产主管 | 主管 |
| **子对象** | Order | WorkOrder | —（最底层） |
| **关键工单标记** | — | — | ✅ isCritical |

## 2. 三层层次模型

### 2.1 层次创建链

```
Plan (CREATED)
  │
  ├── PUBLISH (生产主管触发)
  │   │
  │   ├── 步骤1: 获取 BOM → 工艺流程 → 产线映射
  │   ├── 步骤2: 遍历每条工艺流程
  │   │         ├── 查找空闲产线
  │   │         └── 为每条空闲产线生成 Order {planNo}-{lineNo}
  │   │
  │   └── 结果: N 条 Order (CREATED)
  │
  ▼
Order (CREATED)
  │
  ├── PUBLISH (Plan 联动触发)
  │   │
  │   ├── 步骤1: 获取产线 → 工艺流程
  │   ├── 步骤2: 遍历工艺流程中每道工序
  │   │         ├── 查找具备该工序技能的员工
  │   │         └── 为每个工序×员工生成 WorkOrder
  │   │
  │   ├── 步骤3: 标记首个工单为关键工单 (isCritical=true)
  │   │
  │   └── 步骤4: 自动发布每个工单 (CREATED → RELEASED)
  │
  ▼
WorkOrder (RELEASED)
  │
  ├── START_WORK (员工确认)
  │   记录 actualStartTime
  │
  ├── FINISH_WORK (员工确认)
  │   记录 actualEndTime
  │
  └── 向上联动: 聚合关键工单状态 → Order → Plan
```

### 2.2 各层字段差异

| 字段 | Plan | Order | WorkOrder |
|------|:---:|:---:|:---:|
| 计划数量 | `planNum` | — | — |
| 已完成数量 | `completedNum` (聚合) | — | — |
| 订单数量 | — | `quantity` | — |
| 工单数量 | — | — | `plannedQuantity` |
| 合格品/不良品 | — | `qualifiedProducts`/`defectiveProducts` (聚合) | — |
| 实际数量/报废 | — | — | `actualQuantity`/`scrapQuantity` |
| BOM 关联 | ✅ `bomId` | — | — |
| 产线关联 | — | ✅ `lineNo` | — |
| 工序关联 | — | — | ✅ `processId` |
| 人员关联 | `creatorId`/`publisherId` | — | ✅ `userId` |
| 关键标记 | — | — | ✅ `isCritical` |

## 3. 向下联动（父→子）

### 3.1 Plan → Order 联动（planStateServiceImpl.handleLinkage）

#### PUBLISH — 计划发布下发订单

```
Plan PUBLISH 联动:
  ├── 1. 获取 plan.bomId
  ├── 2. processFlowMapper.getProcessFlowIdList(bomId)
  │      → 获取该 BOM 下所有工艺流程
  ├── 3. 遍历每条工艺流程 flow:
  │      ├── lineMapper.getAvailableLineNoByFlowId(flow.id)
  │      │   → 查找空闲产线
  │      ├── 若无空闲产线 → 跳过此流程
  │      ├── 若有空闲产线:
  │      │   ├── 订单编号 = plan.planNo + "-" + line.lineNo
  │      │   ├── 创建 ProductionOrder:
  │      │   │   · orderNo   = planNo + "-" + lineNo
  │      │   │   · orderName = planName + "-" + lineNo
  │      │   │   · quantity  = planNum (继承计划数量)
  │      │   │   · startTime/endTime = 继承计划时间
  │      │   │   · status = CREATED
  │      │   └── orderMapper.insert(order)
  │      └── 继续下一个工艺流程
  └── 4. 完成
```

#### PAUSE — 计划暂停级联订单

```
Plan PAUSE 联动:
  ├── 1. 查询 plan 下所有 status=RUNNING 的 Order
  ├── 2. 对每个 RUNNING Order:
  │      ├── 查询 Order 下所有 RUNNING 的 WorkOrder
  │      ├── 将每个 WorkOrder status → PAUSED
  │      │   └── auditService.record(WORK_ORDER, ...)
  │      ├── 将 Order status → PAUSED
  │      │   └── auditService.record(ORDER, ...)
  │      └── 下一个 Order
  └── 3. 完成
```

> **实现**: 使用直接 DB 更新方式（`updateWorkOrderStatus` / `updateOrderStatus`），避免通过 Order 服务层触发 `syncPlanStatusFromOrders` 造成循环回写。

#### RESUME — 计划恢复级联订单

```
Plan RESUME 联动:
  ├── 1. 查询 plan 下所有 status=PAUSED 的 Order
  ├── 2. 对每个 PAUSED Order:
  │      ├── 查询 Order 下所有 PAUSED 的 WorkOrder
  │      ├── 将每个 WorkOrder status → RUNNING
  │      │   └── auditService.record(WORK_ORDER, ...)
  │      ├── 将 Order status → RUNNING
  │      │   └── auditService.record(ORDER, ...)
  │      └── 下一个 Order
  └── 3. 完成
```

#### CANCEL_PUBLISH — 取消发布联动

```
Plan CANCEL_PUBLISH 联动:
  ├── 1. 查询 plan 下所有 Order
  ├── 2. 过滤: 仅处理 status ≠ COMPLETED 且 ≠ TERMINATED 的 Order
  ├── 3. 对每个符合条件的 Order:
  │      ├── 查询该 Order 下所有 WorkOrder
  │      ├── 对每个非终态的 WorkOrder:
  │      │   ├── status → TERMINATED
  │      │   └── auditService.record(WORK_ORDER, ..., action=TERMINATE)
  │      ├── Order status → TERMINATED
  │      └── auditService.record(ORDER, ..., action=TERMINATE)
  └── 4. 完成
```

#### TERMINATE — 作废联动

```
Plan TERMINATE 联动:
  ├── 前置: handle() 第5步已校验 plan 下无 RUNNING 状态的 Order
  ├── 1. 查询 plan 下所有 Order
  ├── 2. 过滤: 仅处理非终态 Order (≠ COMPLETED, ≠ TERMINATED)
  ├── 3. 对每个符合条件的 Order:
  │      ├── 查询该 Order 下所有 WorkOrder
  │      ├── 对每个非终态 WorkOrder:
  │      │   ├── status → TERMINATED
  │      │   └── auditService.record(WORK_ORDER, ...)
  │      ├── Order status → TERMINATED
  │      └── auditService.record(ORDER, ...)
  └── 4. 完成
```

### 3.2 Order → WorkOrder 联动（orderStateServiceImpl.handleLinkage）

| Order 动作 | WorkOrder 级联处理 | 工单审计 |
|-----------|-------------------|:---:|
| PUBLISH | 按产线→流程→工序→人员生成 WorkOrder，自动发布 | ✅ |
| CANCEL_PUBLISH | 作废所有非终态工单（CREATED/RELEASED/PAUSED） | ✅ |
| PAUSE | 暂停所有 RUNNING 工单 | ✅ |
| RESUME | 恢复所有 PAUSED 工单 | ✅ |
| TERMINATE | 作废所有非终态工单（CREATED/RELEASED/PAUSED） | ✅ |

## 4. 向上同步（子→父）

### 4.1 WorkOrder → Order（级联聚合）

工单层是最靠近现场的执行层，员工操作驱动：

```
工单状态变更 → workOrderStateServiceImpl.handleLinkage()
  │
  ├── 聚合同一 Order 下所有关键工单(isCritical=true)的状态
  │
  ├── 判定规则:
  │    ├── 任一关键工单 RUNNING → Order → RUNNING (首次)
  │    ├── 任一关键工单 PAUSED 且无 RUNNING → Order → PAUSED
  │    ├── 所有工单 COMPLETED → Order → COMPLETED
  │    └── 无变化 → 保持
  │
  └── 执行: orderMapper.updateOrderStatus(orderNo, newStatus)
            → 触发 orderStateServiceImpl.handleLinkage()
```

### 4.2 Order → Plan（syncPlanStatusFromOrders）

订单状态变更后，通过 `syncPlanStatusFromOrders()` 自动同步计划状态：

```
orderStateServiceImpl.handleLinkage()
  │
  └── planStateServiceImpl.syncPlanStatusFromOrders(planNo)
        │
        ├── 1. 查询 plan 下所有 Order
        ├── 2. O(N) 单次遍历聚合 Order 状态
        │
        ├── 判定规则 (v1.2):
        │    ├── 任一 Order RUNNING → Plan → RUNNING
        │    │
        │    ├── PAUSED 判定（仅"已执行"订单）:
        │    │   · 排除 CREATED/RELEASED 的订单（未开始执行的草稿订单不参与）
        │    │   · 所有已执行订单处于 PAUSED/COMPLETED/TERMINATED
        │    │   · 至少一个 PAUSED
        │    │   · 无 RUNNING
        │    │   → Plan → PAUSED (仅当原状态为 RUNNING)
        │    │
        │    ├── COMPLETED 判定:
        │    │   · 所有订单处于 COMPLETED 或 TERMINATED
        │    │   · 至少一个 COMPLETED (hasCompleted)
        │    │   · 排除全部 TERMINATED 的极端情况
        │    │   → Plan → COMPLETED
        │    │   → 仅汇总 COMPLETED 订单的 quantityProduced
        │    │
        │    └── 无变化 → 保持
        │
        └── 3. planMapper.updatePlanStatus(planNo, newStatus)
              (如 COMPLETED 则同时) planMapper.updatePlanCompletedNum(planNo, num)
```

### 4.3 向上同步汇总表

| 上层对象 | 同步触发点 | 判定依据 | 同步方法 |
|----------|-----------|---------|---------|
| Order | WorkOrder 状态变更 | 关键工单(isCritical=true) 聚合 | `workOrderStateServiceImpl.handleLinkage()` |
| Plan | Order 状态变更 | 所有订单 O(N) 聚合 | `planStateServiceImpl.syncPlanStatusFromOrders()` |

## 5. 各层状态变更详解

### 5.1 Plan 层状态变更

| Action | 可用? | 触发者 | 前置条件 | 级联影响 |
|--------|:---:|------|---------|---------|
| PUBLISH | ✅ | 生产主管 | CREATED | 按工艺流程+产线生成 Order |
| CANCEL_PUBLISH | ✅ | 生产主管 | RELEASED | 作废所有非终态 Order + 工单 |
| PAUSE | ✅ | 生产主管 | RUNNING | 级联暂停 RUNNING Order + 工单 |
| RESUME | ✅ | 生产主管 | PAUSED | 级联恢复 PAUSED Order + 工单 |
| TERMINATE | ✅ | 生产主管 | RELEASED | 作废所有非终态 Order + 工单；前置校验: 无 RUNNING Order |
| START_WORK | ❌ | — | — | 计划层不适用 |
| FINISH_WORK | ❌ | — | — | 计划层不适用 |

### 5.2 Order 层状态变更

| Action | API 可调用? | 触发者 | 前置条件 | 级联影响 |
|--------|:---:|------|---------|---------|
| PUBLISH | ❌ | Plan 联动 | CREATED | 按工艺流程生成 + 发布 WorkOrder |
| CANCEL_PUBLISH | ❌ | Plan 联动 | RELEASED | 作废所有非终态 WorkOrder |
| START_WORK | ❌ | WorkOrder 联动 | RELEASED | 向上同步 Plan |
| PAUSE | ✅ | 主管 | RUNNING | 暂停 RUNNING WorkOrder；向上同步 Plan |
| RESUME | ✅ | 主管 | PAUSED | 恢复 PAUSED WorkOrder；向上同步 Plan |
| FINISH_WORK | ❌ | WorkOrder 联动 | RUNNING | 向上同步 Plan |
| TERMINATE | ✅ | 主管 | RELEASED | 作废非终态 WorkOrder；释放产线；向上同步 Plan |

> **API 白名单**: 控制器内置 `USER_ALLOWED_ACTIONS = [PAUSE, RESUME, TERMINATE]`，其他 Action 直接调用 API 返回 400。

### 5.3 WorkOrder 层状态变更

| Action | API 可调用? | 触发者 | 前置条件 | 级联影响 |
|--------|:---:|------|---------|---------|

| TERMINATE | ✅ | 主管 | RELEASED/PAUSED | 向上同步 Order |

## 6. 状态聚合算法

### 6.1 Order 状态聚合（来自 WorkOrder）

```
输入: orderNo
输出: 新的 Order 状态 (或 null 表示不变化)

算法:
  1. SELECT status FROM work_order
     WHERE order_no = ? AND is_critical = 1   ← 仅关键工单
  2. 收集所有关键工单状态
  3. 如果任一工单 RUNNING → 返回 RUNNING
  4. 如果任一工单 PAUSED 且无 RUNNING → 返回 PAUSED
  5. 如果所有工单 COMPLETED → 返回 COMPLETED
  6. 否则 → 返回 null (不变化)
```

### 6.2 Plan 状态聚合（syncPlanStatusFromOrders）

```
输入: planNo
输出: 新的 Plan 状态 (或 null 表示不变化)

算法 (v1.2):
  1. SELECT * FROM production_order WHERE plan_no = ?
  2. 收集所有订单信息:
     totalOrders     = 总订单数
     runningOrders   = RUNNING 状态的订单
     pausedOrders    = PAUSED 状态的订单
     completedOrders = COMPLETED 状态的订单
     terminatedOrders= TERMINATED 状态的订单
     executedOrders  = 排除 CREATED/RELEASED 的订单 (= 已执行集)
  3. RUNNING 判定:
     if runningOrders.size() > 0 → 返回 RUNNING
  4. PAUSED 判定:
     if runningOrders 为空
        AND pausedOrders 不为空
        AND executedOrders 全是 PAUSED/COMPLETED/TERMINATED
        → 返回 PAUSED (仅当原状态为 RUNNING 时)
  5. COMPLETED 判定:
     if totalOrders = completedOrders + terminatedOrders
        AND completedOrders 不为空 (hasCompleted)
        → 计算 completedNum = SUM(completedOrders.quantityProduced)
        → 返回 COMPLETED
  6. 否则 → 返回 null
```

### 6.3 优先级

```
Plan 状态判定优先级:
  1. RUNNING > 2. PAUSED > 3. COMPLETED

即: 只要有一个订单在运行，计划就是运行态；
否则检查是否应该暂停；
最后检查是否全部完成。
```

## 7. 修改/删除的状态感知约束

### 7.1 修改约束汇总

| 状态 | Plan 可改字段 | Order 可改字段 | WorkOrder 可改字段 |
|------|-------------|--------------|-------------------|
| CREATED | 全部 | 全部 | 全部 |
| RELEASED | planNum/时间/priority/remark | quantity/时间/remark | plannedQuantity/时间/remark |
| RUNNING | ❌ 不允许 | ❌ 不允许 | actualQuantity/scrapQuantity/remark |
| PAUSED | ❌ 不允许 | remark | remark |
| COMPLETED | ❌ 不允许 | ❌ 不允许 | ❌ 不允许 |
| TERMINATED | ❌ 不允许 | ❌ 不允许 | ❌ 不允许 |

### 7.2 删除约束

| 状态 | 三层通用规则 |
|------|-------------|
| CREATED | ✅ 允许删除 |
| 其他所有状态 | ❌ 不允许删除（只允许作废） |

**设计原则**: 一旦已发布（产生了资源分配/级联对象），禁止删除，只能通过 TERMINATE 作废。

### 7.3 作废（TERMINATE）约束

| 层 | 可作废状态 | 前置条件 |
|----|-----------|---------|
| Plan | RELEASED | 无 RUNNING 订单 |
| Order | RELEASED | 无 RUNNING 工单 |
| WorkOrder | RELEASED 或 PAUSED | — |

## 8. 编号生成规则

| 层次 | 编号格式 | 示例 |
|------|---------|------|
| Plan | `PLAN-yyyyMMdd-nnn` | `PLAN-20260608-001` |
| Order | `{planNo}-{lineNo}` | `PLAN-20260608-001-LINE-A01` |
| WorkOrder | `{orderNo}-{processId}-{seq}` | `PLAN-20260608-001-LINE-A01-10-1` |

- Plan 编号由调用方生成传入
- Order 编号由 Plan PUBLISH 联动时自动生成
- WorkOrder 编号由 Order PUBLISH 联动时自动生成
- `seq` 为同一订单+工序下的自增序号，从 1 开始

## 9. 防循环回写机制

### 9.1 问题场景

```
Plan PAUSE → 调用 orderService 逐订单触发 PAUSE
  → 每个 Order PAUSE 触发 syncPlanStatusFromOrders
    → syncPlanStatusFromOrders 可能将 Plan 状态改回
      → 造成循环回写
```

### 9.2 解决方案

Plan 层主动触发的级联联动（PAUSE/RESUME/CANCEL_PUBLISH/TERMINATE）**使用直接 DB 更新**子对象状态：

```java
// planStateServiceImpl.handleLinkage() 中的做法:
// 不通过 orderStateServiceImpl.handle() 逐个触发
// 而是直接更新 DB:
orderMapper.updateOrderStatus(orderNo, newStatus);
workOrderMapper.updateWorkOrderStatus(workOrderNo, newStatus);
auditMapper.addAudit(...);  // 手动写入审计记录
```

### 9.3 正常的向上同步链

```
WorkOrder.handleLinkage()
  → orderMapper.updateOrderStatus()      // 直接 DB 更新 Order 状态
    → planStateService.syncPlanStatusFromOrders()  // 同步 Plan 状态
    → orderStateServiceImpl.handleLinkage()        // Order 级联处理
```

**关键**: 当子对象状态变更触发 `syncPlanStatusFromOrders` 时，Plan 的状态是通过聚合所有 Order 的当前状态来判定的，因此不会因为 Plan 自身的主动操作而产生循环。
