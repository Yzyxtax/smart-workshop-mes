## 1. 设计目标

### 1.1 核心设计思想

状态机系统是整个生产执行引擎的核心，基于以下设计原则：

1. **状态与权限分离**: StateMachine 回答"能否执行"（规则），PermissionPolicy 回答"谁能执行"（权限）
2. **标准化管道**: 所有状态变更遵循完全一致的 8 步处理流程
3. **纯规则引擎**: StateMachine、PermissionPolicy、GatePolicy 均不访问数据库，确保可测试性
4. **审计全覆盖**: 所有状态变更（包括级联触发）写入 status_history 表

### 1.2 组件关系图

```
                    ┌──────────────────┐
                    │   Controller     │
                    │ POST /{res}/{no} │
                    │ /actions/{action}│
                    └────────┬─────────┘
                             │
                             ▼
               ┌─────────────────────────┐
               │  StateService.handle()   │  ← 8步标准管道
               │  ┌─────────────────────┐ │
               │  │ 1. 获取对象+上下文  │ │
               │  │ 2. PermissionPolicy │ │ ← 权限校验
               │  │ 3. StateMachine     │ │ ← 规则校验
               │  │ 4. GatePolicy       │ │ ← 门禁校验
               │  │ 5. 业务验证          │ │

               │  │ 8. handleLinkage    │ │ ← 级联联动
```

## 2. 状态与动作枚举定义

### 2.1 StateEnum — 六种业务状态

```java
public enum StateEnum {
    CREATED,     // 创建 / 草稿
    RELEASED,    // 已发布
    RUNNING,     // 执行中
    PAUSED,      // 已暂停
    COMPLETED,   // 已完成（终态）
    TERMINATED;  // 已作废（终态）

    // 状态转换规则：根据当前状态和动作计算目标状态
    public StateEnum next(ActionEnum action);

    // 是否为终态（COMPLETED 或 TERMINATED）
    public boolean isTerminal();

    // 是否为执行前状态（CREATED 或 RELEASED）
    public boolean isPreExecution();
}
```

### 2.2 ActionEnum — 七种业务动作

```java
public enum ActionEnum {
    PUBLISH,          // 发布 → CREATED → RELEASED
    CANCEL_PUBLISH,   // 取消发布 → RELEASED → CREATED
    PAUSE,            // 暂停 → RUNNING → PAUSED
    RESUME,           // 恢复执行 → PAUSED → RUNNING
    START_WORK,       // 开始作业 → RELEASED → RUNNING
    FINISH_WORK,      // 完成作业 → RUNNING → COMPLETED
    TERMINATE;        // 作废 → 非终态 → TERMINATED
}
```

### 2.3 StateEnum.next() 完整转换表

```java
// StateEnum.next(ActionEnum) 的核心实现逻辑
public StateEnum next(ActionEnum action) {
    switch (this) {
        case CREATED:
            if (action == PUBLISH) return RELEASED;
            break;
        case RELEASED:
            if (action == CANCEL_PUBLISH) return CREATED;
            if (action == START_WORK) return RUNNING;
            if (action == TERMINATE) return TERMINATED;
            break;
        case RUNNING:
            if (action == PAUSE) return PAUSED;
            if (action == FINISH_WORK) return COMPLETED;
            break;
        case PAUSED:
            if (action == RESUME) return RUNNING;
            if (action == TERMINATE) return TERMINATED;
            break;
        default:
            break; // 终态不允许任何转换
    }
    throw new IllegalStateException("非法状态转换");
}
```

## 3. 状态转换规则

### 3.1 Plan（计划）状态-动作矩阵

| 当前状态 ↓ / 动作 → | PUBLISH | CANCEL_PUBLISH | START_WORK | PAUSE | RESUME | FINISH_WORK | TERMINATE |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **CREATED** | ✅ 主管 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **RELEASED** | ❌ | ✅ 主管 | ❌ | ❌ | ❌ | ❌ | ✅ 主管 |
| **RUNNING** | ❌ | ❌ | ❌ | ✅ 主管 | ❌ | ❌ | ❌ |
| **PAUSED** | ❌ | ❌ | ❌ | ❌ | ✅ 主管 | ❌ | ❌ |
| **COMPLETED** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **TERMINATED** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |

**关键差异**: 计划层**不支持** `START_WORK` / `FINISH_WORK` 的人工调用，计划层的 RUNNING/COMPLETED 由订单状态联动上传驱动。

### 3.2 Order（订单）状态-动作矩阵

| 当前状态 ↓ / 动作 → | PUBLISH | CANCEL_PUBLISH | START_WORK | PAUSE | RESUME | FINISH_WORK | TERMINATE |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **CREATED** | ✅ 系统 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **RELEASED** | ❌ | ✅ 系统 | ✅ 工单事实 | ❌ | ❌ | ❌ | ✅ 主管 |
| **RUNNING** | ❌ | ❌ | ❌ | ✅ 主管 | ❌ | ✅ 工单事实 | ❌ |
| **PAUSED** | ❌ | ❌ | ❌ | ❌ | ✅ 主管 | ❌ | ❌ |
| **COMPLETED** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **TERMINATED** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |

**关键差异**: PUBLISH/CANCEL_PUBLISH 仅系统内部触发，API 层面已拦截；START_WORK/FINISH_WORK 由工单状态联动触发。

### 3.3 WorkOrder（工单）状态-动作矩阵

| 当前状态 ↓ / 动作 → | PUBLISH | CANCEL_PUBLISH | START_WORK | PAUSE | RESUME | FINISH_WORK | TERMINATE |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **CREATED** | ✅ 系统 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **RELEASED** | ❌ | ✅ 系统 | ✅ 员工 | ❌ | ❌ | ❌ | ✅ 主管 |
| **RUNNING** | ❌ | ❌ | ❌ | ✅ 员工/主管 | ❌ | ✅ 员工 | ❌ |
| **PAUSED** | ❌ | ❌ | ❌ | ❌ | ✅ 员工/主管 | ❌ | ✅ 主管 |
| **COMPLETED** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **TERMINATED** | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |

**关键差异**: 最靠近现场层，PAUSE/RESUME 支持员工（需 userId 匹配）或主管（跨工单）操作。

## 4. 状态机组件设计

### 4.1 状态机核心实现

三个状态机（`planStateMachine`、`orderStateMachine`、`workOrderStateMachine`）结构完全一致：

```java
@Component
public class XxxStateMachine {

    // 每个状态允许的动作集合
    private static final Map<StateEnum, Set<ActionEnum>> ALLOWED;

    static {
        ALLOWED = new EnumMap<>(StateEnum.class);
        ALLOWED.put(CREATED, Set.of(PUBLISH));
        ALLOWED.put(RELEASED, Set.of(CANCEL_PUBLISH, START_WORK, TERMINATE));
        ALLOWED.put(RUNNING, Set.of(PAUSE, FINISH_WORK));
        ALLOWED.put(PAUSED, Set.of(RESUME, TERMINATE));
        // COMPLETED 和 TERMINATED 为终态，不注册任何允许动作
    }

    /**
     * 校验状态转换的合法性。
     * @param state 当前状态
     * @param action 拟执行动作
     * @throws IllegalStateException 如果转换非法
     */
    public void check(StateEnum state, ActionEnum action) {
        Set<ActionEnum> allowed = ALLOWED.get(state);
        if (allowed == null || !allowed.contains(action)) {
            throw new IllegalStateException(
                String.format("%s 状态 [%s] 不允许执行动作 [%s]",
                    getTypeName(), state, action));
        }
    }
}
```

### 4.2 设计要点

| 特性 | 说明 |
|------|------|
| **无状态 Bean** | 三个状态机均为 `@Component`，无状态，可安全缓存和复用 |
| **纯规则引擎** | 不访问数据库、不涉及用户/角色、不抛 BusinessException |
| **EnumMap 优化** | 使用 `EnumMap<StateEnum, Set<ActionEnum>>` 高效存储规则 |
| **终态零动作** | COMPLETED 和 TERMINATED 不注册任何允许动作，自动拒绝所有操作 |
| **差异化注册** | Plan 不注册 START_WORK/FINISH_WORK；WorkOrder 注册更灵活的 PAUSE/RESUME |

## 5. 状态变更标准管道

### 5.1 8 步处理管道

每个状态服务（`xxxStateServiceImpl`）的 `handle()` 方法遵循完全一致的流程：

```
public Result handle(String bizNo, ActionEnum action, Integer userId) {

    // Step 1: 获取业务对象并构建 StateContext
    T entity = getByNo(bizNo);
    StateContext ctx = StateContext.of(bizNo, action, userId);

    // Step 2: 权限校验（PermissionPolicy）
    permissionPolicy.check(ctx, entity);
    // 失败 → 抛出 SecurityException → 403

    // Step 3: 状态机规则校验（StateMachine）
    stateMachine.check(entity.getStatus(), action);
    // 失败 → 抛出 IllegalStateException → 400

    // Step 4: 门禁校验（GatePolicy）—— 仅在 PUBLISH 等关键动作时触发
    if (action == PUBLISH) {
        gatePolicy.check(entity);
    }
    // 失败 → 抛出 BusinessException → 400

    // Step 5: 业务校验
    // 如 TERMINATE 前检查无 RUNNING 状态关联对象
    validateBusinessRule(entity, action);
    // 失败 → 抛出 BusinessException → 400

    // Step 6: 状态持久化
    StateEnum oldStatus = entity.getStatus();
    StateEnum newStatus = oldStatus.next(action);
    // 按 action 类型差异化更新：
    // START_WORK → 同时锚定 actualStartTime = now
    // FINISH_WORK → 同时锚定 actualEndTime = now
    // PUBLISH → 同时记录 publisherId
    updateStatus(bizNo, newStatus, action);

    // Step 7: 审计记录
    auditService.record(targetType, bizNo, oldStatus, newStatus, userId);

    // Step 8: 级联联动
    handleLinkage(entity, action, userId);
    // 根据 action 类型执行不同的级联策略（见三层生产架构详细设计）

    return Result.success();
}
```

### 5.2 三个状态服务的差异化配置

| 维度 | planStateServiceImpl | orderStateServiceImpl | workOrderStateServiceImpl |
|------|---------------------|----------------------|--------------------------|
| 状态机 | planStateMachine | orderStateMachine | workOrderStateMachine |
| 权限策略 | planPermissionPolicy | orderPermissionPolicy | workOrderPermissionPolicy |
| PUBLISH Gate | ✅ 三道门禁 | ✅ 三道门禁 | ❌（由订单联动触发） |
| TERMINATE 前校验 | 无 RUNNING 订单 | 无 RUNNING 工单 | RELEASED/PAUSED 状态 |
| START_WORK 可用 | ❌ | 仅系统触发 | ✅ 员工手动 |
| FINISH_WORK 可用 | ❌ | 仅系统触发 | ✅ 员工手动 |
| 向下联动 | 创建/暂停/恢复/作废订单 | 创建/暂停/恢复/作废工单 | 向上同步订单状态 |
| 向上联动 | — | syncPlanStatusFromOrders | handleLinkage → 聚合订单状态 |

## 6. 权限策略设计

### 6.1 PermissionPolicy 接口

```java
public interface PermissionPolicy<T> {
    /**
     * 校验操作员是否有权限对指定业务对象执行指定动作。
     * @param ctx 状态上下文（含 action 和 userId）
     * @param entity 业务对象
     * @throws SecurityException 如果权限不足
     */
    void check(StateContext ctx, T entity);
}
```

### 6.2 三层权限策略对比

| 动作 | Plan | Order | WorkOrder |
|------|------|-------|-----------|
| PUBLISH | 生产主管 | 系统联动（不校验） | 系统联动（不校验） |
| CANCEL_PUBLISH | 生产主管 | 系统联动（不校验） | 系统联动（不校验） |
| START_WORK | —（不支持） | 工单事实驱动（不校验） | **员工**（userId 匹配） |
| PAUSE | 生产主管 | 车间主任/生产主管 | **员工**（userId 匹配）**或主管** |
| RESUME | 生产主管 | 车间主任/生产主管 | **员工**（userId 匹配）**或主管** |
| FINISH_WORK | —（不支持） | 工单事实驱动（不校验） | **员工**（userId 匹配） |
| TERMINATE | 生产主管 | 车间主任/生产主管 | 车间主任/生产主管 |

### 6.3 workOrderPermissionPolicy 的复合逻辑

```java
// 伪代码表示权限判断逻辑
if (action == START_WORK || action == FINISH_WORK) {
    // 必须为本工单的派工人员
    if (ctx.userId != entity.getUserId()) throw new SecurityException();
    return;
}
if (action == PAUSE || action == RESUME) {
    // 员工（userId 匹配）或主管角色均可
    if (ctx.userId == entity.getUserId()) return;
    if (hasManagerRole(ctx.userId)) return;
    throw new SecurityException();
}
if (action == TERMINATE) {
    // 仅主管角色
    if (!hasManagerRole(ctx.userId)) throw new SecurityException();
    return;
}
```

### 7.1 GatePolicy 接口

### 7.2 Plan PUBLISH 三道门禁

| 门禁层级 | 校验内容 | SQL 示例 | 失败提示 |
|----------|----------|---------|----------|
| **Capacity Gate**<br>（产线可用性） | 流程图关联的产线中至少有一条为空闲状态 | `SELECT * FROM production_lines WHERE process_flow_id=? AND line_status='空闲'` | "工艺流程没有匹配的可用产线" |
| **Process Gate**<br>（工艺完整性） | BOM 是否有关联的有效工艺流程 | `SELECT * FROM process_flows WHERE bom_id=? AND status='有效'` | "该产品对应的工艺流程未定义或无效" |
| **Skill Gate**<br>（人员可执行性） | 工艺流程中每道工序至少有一名具备技能的人员 | 遍历工序，对每道工序查 skills 表 | "工序「焊接」缺少具备技能的可用员工" |

### 7.3 Order PUBLISH 三道门禁

| 门禁 | 校验 | 失败提示 |
|------|------|----------|
| 产线可用性 | 绑定产线存在且为空闲 | "产线 LINE-A01 不可用或已被占用" |
| 工艺完整性 | 产线是否绑定启用的工艺流程 | "产线 LINE-A01 未绑定有效工艺流程" |
| 人员可执行性 | 工序中每个步骤至少有一名具备技能的可用员工 | "工序[焊接]无可用员工" |

### 7.4 触发条件

| 动作 | Plan 门禁 | Order 门禁 | WorkOrder 门禁 |
|------|:---:|:---:|:---:|
| PUBLISH | ✅ | ✅ | ❌（由订单联动，不独立触发的门禁） |
| 其他动作 | ❌ | ❌ | ❌ |

## 8. 审计服务设计

```java
     * 记录一次状态变更审计。

     * @param newStatus  变更后状态
```

```java
public enum TargetType {
```

### 8.3 审计记录的数据模型

## 9. StateContext 值对象

### 9.1 定义

```java
public class StateContext {
    private final String bizNo;     // 业务对象编号
    private final ActionEnum action; // 执行的动作
    private final Integer userId;   // 操作人 ID

    public static StateContext of(String bizNo, ActionEnum action, Integer userId) {
        return new StateContext(bizNo, action, userId);
    }
}
```

### 9.2 使用场景

`StateContext` 作为不可变值对象，在以下组件间传递：

```
Controller → StateService.handle()
  → PermissionPolicy.check(ctx, entity)
  → GatePolicy.check(entity)        (使用 ctx.action 判断是否触发的门禁)
  → AuditService.record(..., ctx.userId)
  → handleLinkage(entity, ctx.action, ctx.userId)
```
