## 1. 总体规范

### 1.1 协议与数据格式

| 项目 | 规范 |
|------|------|
| 传输协议 | HTTP/1.1 |
| 数据格式 | JSON (application/json) |
| 字符编码 | UTF-8 |
| 基础路径 | `/` (直接挂载在根路径) |

### 1.2 RESTful 约定

```json
// 成功响应
{
  "code": 200,
  "message": "success",
  "data": {}
}

// 错误响应
{
  "code": 400,
  "message": "错误描述信息",
  "data": null
}
```

### 1.3 请求头约定

| 请求头 | 值 | 说明 |
|--------|-----|------|
| Content-Type | application/json | 所有 POST/PUT 请求 |
| Authorization | Bearer \<jwt_token\> | JWT 认证（方式一） |
| token | \<jwt_token\> | JWT 认证（方式二） |
| Accept | application/json | 期望的响应格式 |

## 2. 统一响应格式

### 2.1 Result 通用响应

```java
public class Result<T> {
    private Integer code;    // 业务状态码
    private String message;  // 提示信息
    private T data;          // 响应数据

    public static <T> Result<T> success(T data) {
        return new Result<>(200, "success", data);
    }

    public static <T> Result<T> success(String message, T data) {
        return new Result<>(200, message, data);
    }

    public static <T> Result<T> error(String message) {
        return new Result<>(500, message, null);
    }
}
```

### 2.2 ResultPage 分页响应

```java
public class ResultPage<T> {
    private Long total;      // 总记录数
    private List<T> rows;    // 当前页数据

    public static <T> Result<ResultPage<T>> of(Long total, List<T> rows) {
        return Result.success(new ResultPage<>(total, rows));
    }
}
```

### 2.3 响应状态码

| HTTP 状态码 | code 字段 | 场景 |
|-------------|-----------|------|
| 200 | 200 | 操作成功 |
| 400 | 400 | 业务错误（参数校验失败、状态机拒绝、门禁失败、业务校验失败） |
| 401 | 401 | 未认证（Token缺失/无效/过期） |
| 403 | 403 | 权限不足 |
| 404 | 404 | 资源不存在（可选，部分接口使用 400 统一返回） |
| 500 | 500 | 服务器内部错误 |

## 3. 状态动作模式

### 3.1 标准路径

对于具有状态的业务对象（Plan、Order、WorkOrder），状态变更接口遵循统一的 **RESTful 动作分发模式**：

```
POST /{resource}/{resourceNo}/actions/{action}
```

### 3.2 路径变量

| 变量 | 类型 | 来源 | 说明 |
|------|------|------|------|
| {resource} | String | 路径 | 资源类型（plan/order/workOrder） |
| {resourceNo} | String | 路径 | 资源编号 |
| {action} | ActionEnum | 路径 | 动作枚举，Spring 自动转换 |

### 3.3 设计原则

1. **用户 ID 不通过客户端传入**: `userId` 由服务端从 JWT Token 中自动解析
2. **请求体通常为空**: 状态变更操作本身不需要额外参数（特殊情况如 WorkOrder may add params）
3. **统一管道处理**: 所有 action 走相同的处理管道
4. **异常映射**: Controller 捕获特定异常类型并映射为对应 HTTP 状态码

### 3.4 示例

```
# 发布计划
POST /plan/PLAN-20260608-001/actions/PUBLISH

# 暂停订单
POST /order/PLAN-20260608-001-LINE-A01/actions/PAUSE

# 工单开始作业
POST /workOrder/PLAN-20260608-001-LINE-A01-10-1/actions/START_WORK
```

## 4. HTTP 方法约定

| 方法 | 语义 | 路径示例 | 说明 |
|------|------|---------|------|
| GET | 查询 | /plan | 全部查询或条件分页 |
| GET | 查询 | /plan/{planNo} | 按编号查询单体 |
| POST | 创建 | /plan | 新增资源 |
| POST | 动作 | /plan/{planNo}/actions/{action} | 状态变更 |
| PUT | 更新 | /plan/{planNo} | 按编号更新 |
| DELETE | 删除 | /plan/{planNo} | 按编号删除 |
| DELETE | 批量删除 | /user?ids=1,2,3 | 批量删除（查询参数） |

### 4.1 路径变量与查询参数的分工

| 方式 | 使用场景 | 示例 |
|------|---------|------|
| 路径变量 | 资源主键/唯一标识 | `/plan/{planNo}`, `/user/{id}` |
| 查询参数 | 筛选条件、分页参数、批量ID | `/user?name=张&page=1`, `/user?ids=1,2,3` |

## 5. 路径设计规范

### 5.1 路径命名

| 规则 | 示例 |
|------|------|
| 使用小写 + 短横线 | `/workOrder` 或 `/work-order` |
| 资源名使用名词 | `/user`, `/plan`, `/equipment` |
| 动作使用动词 | `/actions/{action}` |
| 关联查询前缀 | `/order/plan/{planNo}`, `/workOrder/order/{orderNo}` |

### 5.2 路径冲突处理

当固定前缀路径与路径变量路径冲突时，固定前缀路径**必须先声明**：

```
# 正确顺序
GET /workOrder/order/{orderNo}      # 固定前缀 - 先声明
GET /workOrder/user/{userId}         # 固定前缀 - 先声明
GET /workOrder/process/{processId}   # 固定前缀 - 先声明
GET /workOrder/{workOrderNo}         # 路径变量 - 后声明
```

## 6. 错误码体系

### 6.1 错误码定义

| code | HTTP 状态 | 含义 | 场景 |
|------|-----------|------|------|
| 200 | 200 | 成功 | 操作成功执行 |
| 400 | 400 | 业务错误 | 参数错误、状态机拒绝、状态不允许操作、门禁失败 |
| 401 | 401 | 未认证 | Token 缺失、无效、过期 |
| 403 | 403 | 权限不足 | 用户无所需角色/权限 |
| 500 | 500 | 系统异常 | 未预期的服务器内部错误 |

### 6.2 400 错误的细分场景

| 场景 | message 模式 | 触发源 |
|------|-------------|--------|
| 状态机拒绝 | `操作不符合规则: {Resource} 状态 [{status}] 不允许执行动作 [{action}]` | StateMachine.check() → IllegalStateException |
| 门禁失败 | `发布失败：{具体原因}` | GatePolicy.check() → BusinessException |
| 业务校验失败 | `{关联对象}下存在执行中的{子对象}，不允许{操作}` | Service.validateBusinessRule() → BusinessException |
| 状态不允许操作 | `{Resource}处于「{status}」状态，不允许{修改/删除}` | Service CRUD 前校验 → BusinessException |
| 资源不存在 | `未找到编号为 {no} 的{Resource}` | Service getByNo() → BusinessException |
| 系统专属Action | `操作 [{action}] 仅限系统内部触发，不支持直接 API 调用` | Controller USER_ALLOWED_ACTIONS 白名单 |
| 参数错误 | `参数错误: {具体原因}` | Controller 参数验证 |

### 6.3 403 错误的细分场景

| 场景 | message 模式 |
|------|-------------|
| 注解层拦截 | `权限不足` |
| 策略层拦截 | `权限不足: 操作员无权限执行此操作` |
| WorkOrder 员工校验 | `权限不足: 操作员非本工单的派工人员` |

## 7. 认证与安全

### 7.1 JWT Token

| 属性 | 值 |
|------|-----|
| 加密算法 | HS256 (HMAC-SHA256) |
| 有效期 | 24 小时 |
| Claims 内容 | `{"id": userId}` (Integer 类型) |
| 签发时机 | 登录成功时 |
| 传递方式 | `Authorization: Bearer xxx` 或 `token: xxx` 请求头 |

### 7.2 认证流程

```
1. POST /login {userName, password}
2. 验证用户名密码
3. 生成 JWT Token (含 userId)
4. 返回 { id, userName, name, token }

后续请求:
5. 请求头携带 token
6. LoginFilter 拦截，验证 token
7. 通过后放行到后续处理链
```

## 8. 分页规范

### 8.1 分页参数

| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| page | Integer | 否 | 1 | 页码（从1开始） |
| pageSize | Integer | 否 | 10 | 每页记录数 |

### 8.2 分页响应

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "total": 100,
    "rows": [ ... ]
  }
}
```

### 8.3 分页实现

使用 PageHelper 插件：

```java
// Service 层
PageHelper.startPage(page, pageSize);
List<T> list = mapper.selectByCondition(param);
// PageHelper 自动拦截下一个查询，封装分页信息
long total = ((Page) list).getTotal();
return ResultPage.of(total, list);
```

## 9. 日期时间格式

### 9.1 格式定义

| 场景 | 格式 | Java 类型 | 示例 |
|------|------|-----------|------|
| 日期 | yyyy-MM-dd | LocalDate | `2026-06-08` |
| 时间 | yyyy-MM-dd HH:mm:ss | LocalDateTime | `2026-06-08 14:30:00` |
| 时间戳 | yyyy-MM-dd HH:mm:ss | LocalDateTime | `2026-06-08 14:30:00` |

### 9.2 JSON 序列化

实体类中使用 `@JsonFormat` 注解统一格式：

```java
@JsonFormat(pattern = "yyyy-MM-dd")
private LocalDate startTime;

@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
private LocalDateTime createTime;
```

### 9.3 数据库映射

- MyBatis 自动将 `date` → `LocalDate`、`datetime` → `LocalDateTime` 映射
- 需确保 `map-underscore-to-camel-case: true` 开启

## 10. 批量操作规范

## 11. 异常处理映射

### 11.1 Controller 层异常处理

状态变更控制器中统一捕获异常并映射为 HTTP 状态码：

```java
@RestController
public class planController {

    @PostMapping("/{planNo}/actions/{action}")
    public Result handleAction(@PathVariable String planNo,
                                @PathVariable ActionEnum action,
                                HttpServletRequest request) {
        try {
            Integer userId = getUserIdFromToken(request);
            return planStateService.handle(planNo, action, userId);
        } catch (BusinessException e) {
            return Result.error(e.getMessage());           // → 400
        } catch (SecurityException e) {
            return Result.error("权限不足: " + e.getMessage()); // → 403
        } catch (IllegalStateException e) {
            return Result.error("操作不符合规则: " + e.getMessage()); // → 400
        } catch (Exception e) {
            log.error("系统异常", e);
            return Result.error("系统内部执行异常，请联系管理员"); // → 500
        }
    }
}
```

### 11.2 全局异常处理

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        log.error("未捕获异常", e);
        return Result.error("服务器内部错误");
    }
}
```
