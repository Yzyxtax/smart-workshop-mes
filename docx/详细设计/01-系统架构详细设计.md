## 1. 设计目标和原则

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **轻量级 MES** | 面向中小型制造车间的生产执行系统，部署在本地服务器上，无需微服务/容器化架构 |
| **状态驱动** | 生产计划→订单→工单三级业务对象，采用状态机模型驱动，所有状态变更可追溯审计 |
| **RBAC 权限** | 基于角色的访问控制，支持"角色-权限"细粒度分配，权限编码全局唯一 |
| **级联联动** | 父对象状态变更自动向下联动子对象，子对象状态聚合向上同步父对象 |

### 1.2 设计原则

1. **单一职责**: 每个模块仅负责一个业务领域，服务层方法职责清晰
2. **管道模式**: 状态变更遵循"权限校验→状态机校验→门禁校验→业务校验→持久化→审计→联动"七步标准管道
3. **不可逆终态**: COMPLETED 和 TERMINATED 为终态，进入终态后不可再执行任何状态变更
4. **审计全覆盖**: 所有状态变更必须记录审计日志（status_history 表），包括从上级层触发的级联变更
5. **状态感知**: 修改/删除操作感知对象当前状态，不同状态有不同的字段修改范围和操作约束

### 2.1 技术栈

```
┌──────────────────────────────────────────────────┐
│                   前端 (Spring MVC + JSON)          │
```

## 4. 分层架构

### 4.1 五层分层模型

```
┌────────────────────────────────────────────┐
│  接入层 (Filter / Interceptor)              │
│  LoginFilter → PermissionInterceptor       │
│  职责: JWT 认证、权限验证                     │
├────────────────────────────────────────────┤
│  控制器层 (Controller)                      │
│  职责: 参数绑定、路径变量转换、异常映射          │
│  输入: JSON Body / Path Variable / Query Param │
│  输出: Result<T> / ResultPage<T>             │
├────────────────────────────────────────────┤
│  服务层 (Service)                           │
│  职责: CRUD 编排、状态感知修改删除、分页查询     │
│  状态服务: 管道模式处理状态变更                 │
├────────────────────────────────────────────┤
│  领域层 (smart-workshop-production)          │
│  职责: 状态机规则、权限策略、门禁校验、审计       │
│  不依赖数据库，纯规则校验                       │
├────────────────────────────────────────────┤
│  持久层 (Mapper + XML)                      │
│  职责: MyBatis 数据访问、动态 SQL              │
│  约定: 接口在 mapper/，XML 在 resources/       │
└────────────────────────────────────────────┘
```

### 4.2 各层间通信机制

| 通信方向 | 方式 | 示例 |
|----------|------|------|
| Controller → Service | 接口注入 | `@Autowired planService` |
| Service → Mapper | 接口注入 | `@Autowired planMapper` |
| Service → Production | 接口注入 | `@Autowired planStateMachine` |
| Production → Common | 直接依赖 | `BusinessException` |
| Server → Pojo | 直接依赖 | `Plan`, `WorkOrder` 实体 |

### 4.3 跨层调用约束

- Controller 层**不直接**访问 Mapper 层，必须通过 Service
- 控制器**不接收** userId 参数（状态变更场景），由服务端从 JWT Token 解析
- Production 模块**不访问**数据库，仅提供纯规则校验
- DTO 仅用于 API 请求体，VO 用于组装嵌套返回结果

## 6. 请求处理流程

### 6.1 完整请求链路

每一个 HTTP 请求按以下顺序执行：

```
HTTP 请求
  │
  ▼
1. LoginFilter (@WebFilter "/*")
   ├── /login 请求 → 放行
   ├── 其他请求 → 提取 JWT Token（Authorization: Bearer 或 token 头）
   │            → JwtUtils.parseToken() 验证
   │            → 失败 → 返回 401
   │            → 成功 → 放行
  │
  ▼
2. PermissionInterceptor (注册到 /**)
   ├── 排除路径: /login, /error, /permission/**, /role/**, /auth/**
   ├── 检查处理器方法或声明类上是否有 @RequirePermission 注解
   │   ├── 无注解 → 放行
   │   └── 有注解 → 解析 Token 获取 userId
   │              → AuthService.getUserPermissionCodes(userId)
   │              → 按 AND/OR 逻辑校验所需权限
   │              → 失败 → 返回 403
   │              → 成功 → 放行
  │
  ▼
3. Controller
   └── 路径变量转换、参数绑定
      └── 调用 Service
  │
  ▼
4. Service
   ├── CRUD 操作 → Mapper
   └── 状态变更 → 管道模式
                    ├── PermissionPolicy
                    ├── StateMachine
                    ├── GatePolicy
                    ├── 业务校验
                    ├── 持久化
                    ├── AuditService
                    └── handleLinkage (级联)
  │
  ▼
5. Mapper (MyBatis)
   └── SQL 执行 → MySQL
  │
  ▼
6. HTTP 响应
   └── Result<T> / ResultPage<T> (JSON)
```

### 6.2 异常处理映射

| 异常类型 | HTTP 状态码 | 控制器处理 |
|----------|-------------|-----------|
| `BusinessException` | 400 | `@ExceptionHandler` → `Result.error(msg)` |
| `SecurityException` | 403 | `@ExceptionHandler` → `Result.error("权限不足: " + msg)` |
| `IllegalStateException` | 400 | `@ExceptionHandler` → `Result.error("操作不符合规则: " + msg)` |
| 未知异常 | 500 | `GlobalExceptionHandler` → `Result.error("系统内部执行异常")` |
| Token 验证失败 | 401 | `LoginFilter` → 直接返回 401 JSON |

## 7. 核心设计决策

### 7.1 状态机与权限策略分离

**设计决策**: 状态机（StateMachine）和权限策略（PermissionPolicy）是两个独立组件。

- **StateMachine**: 纯规则引擎，判断 "当前状态下是否允许执行此动作"（不涉及用户/角色）
- **PermissionPolicy**: 权限判断，回答 "谁可以执行此动作"

**理由**: 职责单一，状态迁移规则变更不影响权限控制，权限策略变更不影响状态流转规则。

### 7.2 管道模式标准化状态变更

**设计决策**: 所有状态变更服务（planStateServiceImpl、orderStateServiceImpl、workOrderStateServiceImpl）遵循完全一致的 8 步处理管道。

**理由**: 确保每个状态变更都经过完整的权限、规则、门禁、审计、联动处理链，避免遗漏。

### 7.3 级联采用直接 DB 更新

**设计决策**: 上级层主动触发的级联操作使用直接 DB 更新方式（如 `updateWorkOrderStatus`），避免通过子级服务层触发递归回写。

**理由**: 避免循环回写（如 Plan PAUSE → Order PAUSE → Plan syncPlanStatusFromOrders 再被触发），保证级联操作的单向性和可控性。

### 7.4 终态不可逆

**设计决策**: COMPLETED 和 TERMINATED 为终态，进入后禁止任何状态变更、修改、删除。

**理由**: 保证数据最终一致性，已完成的业务事实不可回退。

### 7.5 权限编码预定义

**设计决策**: 权限编码由开发阶段通过数据库初始化脚本定义，**不提供运行时 API 创建/修改/删除接口**。

**理由**: 权限编码与代码中的 `@RequirePermission` 注解强绑定，运行时变更会导致代码一致性破坏。

### 7.6 编号由调用方生成

**设计决策**: Plan 编号（`PLAN-yyyyMMdd-nnn`）和 Order 编号（`{planNo}-{lineNo}`）由调用方/系统逻辑生成传入，后端不自增生成。

**理由**: 编号中内嵌业务含义（日期、产线），便于人工识别和追溯。

## 8. 包结构与命名规范

### 8.1 包路径

所有 Java 代码位于 `com.xtax` 包下。

### 8.2 文件命名

| 类型 | 命名风格 | 示例 |
|------|----------|------|
| 实体类 | PascalCase | `Plan.java`, `WorkOrder.java` |
| DTO | PascCase + 后缀 | `AssignRoleDTO.java`, `WorkOrderDTO.java` |
| VO | PascCase + 后缀 | `RoleVO.java`, `UserPermissionVO.java` |
| 控制器 | camelCase (旧) / PascalCase (新) | `planController.java`, `RoleController.java` |
| 服务接口 | camelCase (旧) / PascalCase (新) | `planService.java`, `AuthService.java` |
| 服务实现 | 接口名 + Impl | `planServiceImpl.java`, `AuthServiceImpl.java` |
| Mapper 接口 | camelCase (旧) / PascalCase (新) | `planMapper.java`, `PermissionMapper.java` |
| XML 映射 | 与 Mapper 接口同名 | `planMapper.xml` |
| 状态机 | camelCase | `planStateMachine.java` |
| 权限策略 | camelCase | `planPermissionPolicy.java` |

> **注意**: 命名风格混合是由于项目发展历史。新模块（RBAC）使用 PascalCase，旧模块使用 camelCase。扩展现有模块时遵循该模块的既有风格。

## 9. 技术选型理由

| 技术 | 理由 |
|------|------|
| **Spring Boot 3.5.7** | 最新稳定版，Jakarta 迁移，支持 Java 21 |
| **MyBatis（非 JPA）** | 复杂查询可控，动态 SQL 灵活，适合 MES 系统中多表联合查询和状态聚合场景 |
| **MySQL InnoDB** | 事务支持、外键级联、枚举类型，适合 MES 的数据一致性要求 |
| **JWT 无状态认证** | 适合前后端分离架构，无需服务端 session 存储 |
| **Maven 多模块** | 清晰的依赖管理，pojo/common/production 三个底层模块可独立演进 |
| **Lombok** | 减少样板代码（getter/setter/constructor/logger），提升开发效率 |
| **PageHelper** | 透明的 MyBatis 分页实现，无需修改 SQL |

## 10. 附录: 架构约束与边界

### 10.1 架构约束

1. **单体部署**: 系统以单体应用方式部署，不使用微服务架构
2. **无分布式事务**: 所有数据库操作在单库内完成，依赖 MySQL 事务保证一致性
3. **无缓存层**: 当前权限校验每次直接查询数据库（三表 JOIN），Redis 缓存为待实现功能
4. **无消息队列**: 级联联动通过同步方法调用完成，不涉及异步消息

### 10.2 已知技术债务

| 序号 | 待办项 | 优先级 |
|------|--------|--------|
| 1 | 生产域 CRUD 接口权限保护（添加 @RequirePermission） | 高 |
| 2 | 权限缓存实现（Redis/Caffeine） | 中 |
| 3 | 命名风格统一（camelCase → PascalCase） | 低 |
| 4 | User 实体补充 `team_no` 字段映射 | 低 |
| 5 | 单元测试覆盖率提升 | 中 |
