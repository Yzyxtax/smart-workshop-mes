## 1. 安全架构概述

### 1.1 多层安全防护

```
HTTP 请求
  │
  ▼
┌─────────────────────────────────────────────┐
│  第一层: 身份认证 (LoginFilter)               │
│  职责: 验证 JWT Token                        │
│  放行: /login                               │
│  失败: 401 Unauthorized                      │
├─────────────────────────────────────────────┤
│  第二层: 注解级权限 (PermissionInterceptor)      │
│  职责: @RequirePermission 注解权限校验         │
│  失败: 403 Forbidden                         │
├─────────────────────────────────────────────┤
│  第三层: 策略级权限 (PermissionPolicy)          │
│  职责: 服务层按 action 类型+业务对象权限校验     │
│  失败: 403 Forbidden (SecurityException)      │
├─────────────────────────────────────────────┤
│  第四层: CORS 跨域 (CorsConfig)               │
│  职责: 允许合法的跨域请求                        │
└─────────────────────────────────────────────┘
```

### 1.2 安全组件一览

| 组件 | 类型 | 作用域 | 作用 |
|------|------|--------|------|
| `LoginFilter` | Servlet Filter | `/*` | JWT 认证 |
| `PermissionInterceptor` | HandlerInterceptor | `/**` | 注解级权限校验 |
| `PermissionPolicy` (接口) | Service | 状态服务内部 | 策略级权限校验 |
| `CorsConfig` | @Configuration | 全局 | 跨域配置 |
| `JwtUtils` | 工具类 | 全局 | JWT 生成/验证 |

### 2.3 Token 传递方式

支持两种传递方式（LoginFilter 同时检查两个请求头）：

```
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

或

```
token: eyJhbGciOiJIUzI1NiJ9...
```

### 2.4 Token 解析流程

```
1. 从请求头获取 token 字符串
   ├── 优先从 "token" 头获取
   └── 其次从 "Authorization" 头提取（去掉 "Bearer " 前缀）

2. JwtUtils.parseToken(token)
   ├── 验证签名（HMAC-SHA256）
   ├── 验证过期时间
   └── 提取 Claims Map {"id": userId}

3. 获取 userId = (Integer) claims.get("id")

4. 对于权限校验场景:
   └── AuthService.getUserPermissionCodes(userId)
        → 查询用户的所有权限编码
```

## 3. LoginFilter 过滤器设计

### 3.1 过滤器实现

```java
@WebFilter("/*")
public class LoginFilter implements Filter {

    // 放行路径
    private static final String LOGIN_PATH = "/login";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) {
        HttpServletRequest req = (HttpServletRequest) request;
        HttpServletResponse resp = (HttpServletResponse) response;

        // 1. 放行 /login 路径
        if (LOGIN_PATH.equals(req.getServletPath())) {
            chain.doFilter(request, response);
            return;
        }

        // 2. 提取 Token
        String token = extractToken(req);
        if (token == null || token.isEmpty()) {
            sendUnauthorized(resp, "未提供认证令牌");
            return;
        }

        // 3. 验证 Token
        try {
            Map<String, Object> claims = JwtUtils.parseToken(token);
            // Token 有效，放行到后续链路
            chain.doFilter(request, response);
        } catch (Exception e) {
            sendUnauthorized(resp, "Token 无效或已过期");
        }
    }

    private String extractToken(HttpServletRequest request) {
        // 先尝试 "token" 头
        String token = request.getHeader("token");
        if (token != null && !token.isEmpty()) return token;

        // 再尝试 "Authorization: Bearer xxx" 头
        String auth = request.getHeader("Authorization");
        if (auth != null && auth.startsWith("Bearer ")) {
            return auth.substring(7);
        }
        return null;
    }
}
```

### 3.2 401 响应格式

```json
{
  "code": 401,
  "message": "未提供认证令牌",
  "data": null
}
```

或

```json
{
  "code": 401,
  "message": "Token 无效或已过期",
  "data": null
}
```

### 3.3 过滤器注册

LoginFilter 通过 `@WebFilter("/*")` 注解自动注册，无需额外的配置类。

## 4. 登录流程设计

### 4.1 接口设计

| 项目 | 说明 |
|------|------|
| 路径 | `POST /login` |
| 请求体 | `{"userName": "admin", "password": "123456"}` |
| 响应 | `{"code": 200, "data": {"id": 1, "userName": "admin", "name": "管理员", "token": "eyJ..."}}` |

### 4.2 登录流程

```
1. 接收请求: { userName, password }

2. userMapper.getByUserName(userName)
   └── 根据用户名查询用户记录

3. 验证:
   ├── 用户不存在 → 返回 400 "用户名或密码错误"
   ├── 密码不匹配 → 返回 400 "用户名或密码错误"
   └── 验证通过 → 继续

4. 生成 Token:
   └── JwtUtils.generateToken({"id": user.id})

5. 返回 LoginInfo:
   └── { id, userName, name, token }
```

### 4.3 LoginInfo 响应对象

```java
public class LoginInfo {
    private Integer id;       // 用户ID
    private String userName;  // 用户名
    private String name;      // 姓名
    private String token;     // JWT Token
}
```

> **注意**: 登录接口当前**不返回**角色/权限信息。如需获取角色和权限，前端需在登录后单独调用 `GET /auth/permissions`。

### 4.4 密码安全

| 现状 | 风险 | 建议 |
|------|------|------|
| 密码以明文存储在 `users.password` 字段中 | 数据库泄露将直接暴露所有用户密码 | 迁移至 BCrypt 加密存储 |

**改进方案**:

```java
// 登录时验证（BCrypt）
if (!BCrypt.checkpw(inputPassword, user.getPassword())) {
    throw new BusinessException("用户名或密码错误");
}

// 创建用户时加密（BCrypt）
user.setPassword(BCrypt.hashpw(rawPassword, BCrypt.gensalt()));
```

## 5. 权限拦截器设计

### 5.1 拦截器注册

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Autowired
    private PermissionInterceptor permissionInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(permissionInterceptor)
                .addPathPatterns("/**")
                .excludePathPatterns(
                    "/login",
                    "/error",
                    "/permission/**",
                    "/role/**",
                    "/auth/**"
                );
    }
}
```

### 5.2 排除路径说明

| 排除路径 | 原因 |
|----------|------|
| `/login` | 认证入口，无 Token 也能访问 |
| `/error` | Spring Boot 错误处理端点 |
| `/auth/**` | 权限查询自身的 API（仅需 JWT 认证，无需权限注解） |
| `/permission/**` | 权限查询，GET 请求自动放行（写操作通过 `@RequirePermission` 保护） |
| `/role/**` | 角色查询，GET 请求自动放行（写操作通过 `@RequirePermission` 保护） |

### 5.3 拦截器的放行逻辑

```java
// PermissionInterceptor 中:
// 1. /role/** 和 /permission/** 的 GET 请求 → 自动放行
if (isGetRequest && (path.startsWith("/role") || path.startsWith("/permission"))) {
    return true;
}

// 2. 排除路径（已在 WebConfig 中注册）→ 不经过拦截器

// 3. 无 @RequirePermission 注解的处理器 → 放行
if (annotation == null) {
    return true;
}

// 4. 有注解 → 执行权限校验
```

### 5.4 权限判定的 AND/OR 逻辑

```java
// AND: 用户必须拥有所有声明的权限
if (annotation.logical() == Logical.AND) {
    for (String perm : annotation.value()) {
        if (!userPermissions.contains(perm)) {
            return false; // 缺少任意一个 → 403
        }
    }
    return true;
}

// OR: 用户拥有任意一个即可
if (annotation.logical() == Logical.OR) {
    for (String perm : annotation.value()) {
        if (userPermissions.contains(perm)) {
            return true; // 命中任意一个 → 放行
        }
    }
    return false; // 一个都没有 → 403
}
```

## 6. 双层权限控制

### 6.1 设计原因

系统有两类权限校验场景，需要不同的控制方式：

| 场景 | 特征 | 控制方式 |
|------|------|---------|
| **CRUD 操作** | 每个操作有明确的权限编码 | `@RequirePermission` 注解 + `PermissionInterceptor` |
| **状态变更** | 同一 API 路径对应多种 action，权限规则取决于 action 类型和业务对象 | `PermissionPolicy` 策略接口 + 服务层内嵌 |

### 6.2 注解层（声明式）

适用场景：CRUD 的增删改操作

```java
@RequirePermission("SYS_USER_CREATE")
@PostMapping("/user")
public Result addUser(@RequestBody User user) { ... }

@RequirePermission("SYS_USER_DELETE")
@DeleteMapping("/user")
public Result deleteUsers(@RequestParam List<Integer> ids) { ... }
```

### 6.3 策略层（编程式）

适用场景：状态变更的细粒度权限控制

```
POST /plan/{planNo}/actions/{action}
  → planController.handleAction()
    → planStateServiceImpl.handle()
      → planPermissionPolicy.check(ctx, plan)  // 策略层校验
```

策略层优势：
- 同一 API 可处理多种 action，权限规则随 action 动态变化
- 支持基于业务对象属性的权限判断（如 WorkOrder 的 userId 匹配）
- 权限规则变更不影响 Controller 层代码

### 6.4 安全状态现状

| 接口类别 | 当前保护状态 |
|----------|-------------|
| RBAC 管理写操作 | ✅ `@RequirePermission` 已应用 |
| 生产域状态变更 | ✅ `PermissionPolicy` 已应用 |
| 生产域 CRUD 写操作 | ⚠️ 仅 `LoginFilter`，待补全 `@RequirePermission` |

## 8. 安全风险与改进建议

### 8.1 已识别风险

| 风险等级 | 问题 | 影响 |
|:---:|------|------|
| 🔴 高 | 密码明文存储 | 数据库泄露 → 所有用户密码暴露 |
| 🟡 中 | 生产域 CRUD 写操作无权限注解 | 任意登录用户可修改/删除计划/订单/设备 |
| 🟡 中 | JWT 密钥硬编码在源码中 | 密钥泄露风险，无法动态更换 |
| 🟡 中 | 无请求频率限制 | 可能被暴力破解登录 |
| 🟡 中 | 无 Token 刷新机制 | 24h 后必须重新登录，体验不佳 |
| 🟢 低 | 权限直接查 DB（无缓存） | 每次请求执行三表 JOIN，性能影响可控但可优化 |

### 8.2 改进优先级建议

| 优先级 | 改进项 | 说明 |
|:---:|------|------|
| P0 | 密码迁移至 BCrypt | 立即修复最高安全风险 |
| P1 | 生产域 CRUD 补全 `@RequirePermission` | 已规划的待办项 |
| P1 | JWT 密钥移至外部配置 | 使用 `application.yml` 或环境变量 |
| P2 | 权限缓存 (Caffeine/Redis) | 减少数据库查询 |
| P2 | 登录失败次数限制 | 防止暴力破解 |
| P3 | Token 刷新机制 | 提升用户体验 |
| P3 | 输入参数校验增强 | SQL 注入防护（MyBatis `#{}` 已做基础防护） |
