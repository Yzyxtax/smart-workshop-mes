## 1. 引言

### 1.1 编写目的

本文档描述"智能车间轻量级 MES 系统"后端的功能性与非功能性需求，作为开发、测试、验收以及后续维护升级的统一基线。本文档面向以下读者：

- 项目经理与产品负责人：理解系统目标、范围与交付边界。
- 开发与测试工程师：作为接口、业务规则与质量属性的实现依据。
- AI/LLM 应用开发者：理解 MES 系统对外暴露的能力边界、工具调用规范与安全约束。
- 运维与部署人员：明确系统的运行环境、外部依赖（含 LLM API）与部署约束。
- 客户/业务方：确认本系统是否满足车间生产管理的实际诉求。

### 1.2 项目背景

本系统面向中小型制造企业的车间现场管理，解决从**生产计划 → 生产订单 → 工单**的三级生产调度与执行跟踪问题，并提供物料清单（BOM）、工艺流程、工序、工作步骤、设备、产线、班组、技能矩阵等主数据维护，以及基于角色的访问控制（RBAC）。系统采用轻量级单体多模块架构，便于中小型企业部署与维护。此外，系统通过接入大语言模型（LLM），提供自然语言交互式操作能力，用户无须记忆复杂的操作路径即可通过对话完成常见的数据查询与写入。

### 1.3 文档范围

本文档覆盖：

- 系统在生产、主数据、权限、安全、审计等领域的功能需求；
- AI 智能交互功能的需求，包括自然语言操作、意图识别、工具调用与会话管理；
- 系统对外提供的 RESTful API 与对内依赖的软件、硬件、通信接口需求（含 LLM API）；
- 性能、可靠性、安全、可维护性、可扩展性等非功能性指标。

不在本文档范围内：

- 前端 Web/移动端 UI 详细设计（含 AI 对话窗口的视觉与交互设计）；
- 第三方 ERP/WMS 集成的具体协议（仅在外部接口章节做开放性描述）；
- 企业级容灾、多活、跨地域部署等高阶运维方案；
- LLM 模型本身的训练、微调或私有化部署方案。

## 2. 综合描述

### 2.1 产品定位

本系统是面向中小型制造企业的**轻量级 MES 后端服务**，对外以 RESTful API 形式提供能力。系统采用 Spring Boot 单体多模块架构，使用 MySQL 作为主数据存储，JWT 作为认证机制，RBAC 作为授权模型，并以"状态机 + 级联联动 + 审计追踪"作为核心业务骨架。

### 2.2 产品功能概述

系统的核心能力可归纳为四大领域：

| 领域 | 核心能力 |
|------|----------|
| **生产核心域** | 生产计划（Plan）、生产订单（Order）、工单（WorkOrder）的全生命周期管理；三层级联状态联动；状态变更审计 |
| **主数据域** | BOM、工艺流程、工序、工作步骤、设备、产线、班组（团队）、产线编组、技能矩阵的维护 |
| **支撑域** | RBAC 用户/角色/权限管理；JWT 登录与无状态鉴权；接口级权限校验；业务门禁（产能/工艺/技能） |
| **智能交互域** | AI 自然语言对话式操作；LLM 意图识别与工具调用；多轮会话管理；AI 操作安全沙箱与审计 |

### 2.3 用户类与特性

系统按照数据库 `role` 表中的实际定义，划分为以下 6 类用户角色：

| 用户类 | 角色编码（`role_code`） | 角色描述 | 主要操作 |
|--------|------------------------|----------|----------|
| 生产主管 | `ROLE_PROD_MANAGER` | 负责生产计划的制定、发布和管理，监督生产进度，协调生产资源 | 创建/发布/取消发布/暂停/恢复/作废生产计划与生产订单；查看生产进度与审计记录 |
| 工艺工程师 | `ROLE_PROCESS_ENGINEER` | 负责工艺流程设计、BOM 管理、工序定义和工艺参数优化 | 维护 BOM、工艺流程、工序、工作步骤、设备等主数据；定义工艺参数与质量控制点 |
| 人事主管 | `ROLE_HR_MANAGER` | 负责人员管理、岗位分配、技能管理和权限分配 | 维护用户、角色、班组人员；分配用户-角色与角色-权限；维护员工技能矩阵 |
| 车间主任 | `ROLE_WORKSHOPS_DIRECTOR` | 负责车间整体管理、生产调度、设备管理和安全管理 | 暂停/恢复/作废生产订单与工单；监控产线状态；调度车间资源 |
| 班组长 | `ROLE_TEAM_LEADER` | 负责班组管理、工单分派、进度跟踪和质量控制 | 维护班组信息；分派/监督工单执行；跟踪生产进度与质量数据 |
| 工人 | `ROLE_WORKER` | 负责执行生产任务、报工和设备操作 | 接收工单；执行 START_WORK/FINISH_WORK；上报实际产量、合格品与不良品数量 |

> 角色编码、角色名称与描述均来源于数据库 `role` 表的真实数据；权限编码采用 `模块_功能` 命名格式（例如 `SYS_ROLE_MANAGE`、`SYS_USER_MANAGE`、`SYS_PERMISSION_ASSIGN` 等），由 `permission` 表统一维护，并通过 `role_permission` 表与角色绑定。所有角色均可使用 AI 交互功能，AI 执行操作的权限范围与该用户的 RBAC 角色权限一致。

### 2.4 运行环境

| 项 | 要求 |
|----|------|
| 操作系统 | 主流 Linux / Windows Server / macOS（开发环境） |
| JDK | Java 21 及以上 |
| 应用服务器 | Spring Boot 3.5.7 内嵌 Tomcat |
| 数据库 | MySQL 8.x（支持 JSON 类型） |
| 构建工具 | Maven ≥ 3.9 |
| 客户端 | 任意支持 HTTP/HTTPS + JSON 的 Web 浏览器或移动端 |

### 2.5 设计与实现约束

- **单体多模块**：Maven 父子工程（pojo / common / production / server）部署为单个可执行 JAR。
- **包名规范**：所有代码位于 `com.xtax` 包下，按层级分包。
- **状态语义**：所有具有生命周期的业务对象使用统一的 `StateEnum`（CREATED/RELEASED/RUNNING/PAUSED/COMPLETED/TERMINATED）和 `ActionEnum`（PUBLISH/CANCEL_PUBLISH/PAUSE/RESUME/START_WORK/FINISH_WORK/TERMINATE）。
- **状态机纯净性**：状态机仅做规则校验，不访问数据库、不涉及权限。
- **无状态服务**：服务端不保存 Session，认证完全基于 JWT。
- **数据库一致性**：跨表写操作通过事务保证；级联联动通过 Service 层编排实现。
- **物理分页**：列表查询统一采用 PageHelper 物理分页，避免内存全量加载。
- **JSON 半结构化字段**：技能矩阵 `process_matrix` 等字段以 JSON 存储，通过 `JsonToMapTypeHandler` 转换。

### 2.6 假设与依赖

- 假设企业内部网络可靠，且服务器与数据库间网络延迟稳定。
- 假设 MySQL 已部署且可远程访问，且配置了合理的备份策略。
- 假设客户端在每次请求中正确携带 `Authorization: Bearer <token>` 或 `token` 头。
- 假设车间一线终端具备稳定网络连接以上报工单进度。
- 假设 LLM API 服务（如 Claude API、OpenAI API 或私有化部署模型）可通过网络访问，且延迟与可用性满足交互需求。

- 概要设计与详细设计文档（详见 `docx/概要设计说明书.md`、`docx/详细设计/`）。

## 3. 外部接口需求

### 3.1 用户接口（UI）

本系统为后端服务，**不直接提供 UI**，但需配合前端 Web/移动端使用。对前端的接口要求：

- 前端 SHALL 通过 HTTP/HTTPS 调用本系统 RESTful API。
- 前端 SHALL 在登录后将 JWT 持久化（推荐 LocalStorage 或安全 Cookie），并在后续请求中通过 `Authorization: Bearer <token>` 头携带。
- 前端 SHOULD 根据接口返回的 `code` 字段（1=成功，0=失败）做统一错误处理；HTTP 状态码 401/403/400/500 应分别映射为登录失效、权限不足、业务错误、服务器异常。
- 前端 SHOULD 支持中英文混排展示；状态值（如 `CREATED`/`RUNNING`）应进行本地化显示。

### 3.2 硬件接口

- 系统对硬件无直接控制需求；如需对接 PLC、扫码枪、RFID、看板大屏等设备，由前端或独立网关服务负责，本系统通过 RESTful API 接收上报数据（如开始工单、完成工单、上报数量）。
- 服务器硬件最低建议：2 核 CPU / 4 GB 内存 / 50 GB 磁盘。

### 3.3 软件接口

| 接口 | 类型 | 说明 |
|------|------|------|
| MySQL 8.x | 数据库 | JDBC 连接，承载所有持久化数据；支持 JSON 列类型 |
| MyBatis + PageHelper | ORM | 使用 XML 映射 + 注解；统一物理分页 |
| Spring Boot 3.5.7 | 框架 | 内嵌 Tomcat，单 JAR 部署 |
| jjwt 0.12.6 | JWT 库 | HS256 签名；24 小时有效期；密钥由应用配置管理 |
| Logback / SLF4J | 日志 | 控制台 + 滚动文件输出（详见非功能需求） |
| Lombok / Jackson | 工具库 | 实体注解简化、JSON 序列化 |
| LLM API（Claude / OpenAI / 私有部署模型等） | AI 服务 | RESTful API 调用，支持 Function Calling / Tool Use 能力；可配置多提供商与模型；支持 HTTP 代理与超时控制 |

> 系统对外开放的可选集成点（未在当前版本实现）：ERP/WMS 数据交换接口、消息队列上报接口、报表系统数据导出接口。这些 MAY 在后续版本通过新增 Controller 提供。

### 3.4 通信接口

- **协议**：HTTP/1.1，生产环境 SHOULD 启用 HTTPS。
- **数据格式**：请求/响应均为 JSON（UTF-8 编码）；MIME 类型 `application/json`。
- **API 风格**：RESTful，兼具状态-动作模式（`POST /{resource}/{id}/actions/{action}`）。
- **认证方式**：JWT（HS256），通过 `Authorization: Bearer <token>` 头或自定义 `token` 头传递。
- **跨域**：开发/测试环境通过 `CorsConfig` 全部放通（允许所有来源、方法、请求头，预检缓存 3600 秒，支持凭证）；生产环境 SHOULD 收紧来源白名单。
- **统一响应**：
  - 单对象/操作：`{"code": 1|0, "message": "...", "data": {...}}`
  - 分页：`{"total": N, "rows": [...]}`
- **错误码**：HTTP 状态码 + 业务 `code` 双重表达；常见状态码 200/400/401/403/500。
- **分页参数**：`page`、`pageSize`，由 PageHelper 统一拦截。

## 4. 系统特性

> 本章按"特性 = 一组紧耦合的相关需求"组织。每个特性包含：4.x.1 描述与优先级、4.x.2 触发与响应、4.x.3 功能需求（编号化）。

### 4.1 特性：身份认证与登录（JWT）

#### 4.1.1 描述与优先级
**优先级：高**。系统所有功能性接口（除登录、错误页、白名单接口）必须通过 JWT 认证。

#### 4.1.2 触发与响应
- **触发**：客户端调用 `POST /login` 提交用户名/密码；或后续请求未携带有效 token。
- **响应**：登录成功签发 JWT；非法/过期 token 返回 HTTP 401。

#### 4.1.3 功能需求
- **REQ-AUTH-001**：系统 SHALL 提供 `POST /login` 接口，接受用户名与密码，校验通过后签发 JWT。
- **REQ-AUTH-002**：JWT SHALL 使用 HS256 算法签名，有效期 24 小时；Claims SHALL 包含至少 `id`（用户 ID，Integer）。
- **REQ-AUTH-003**：系统 SHALL 通过 `LoginFilter`（`@WebFilter "/*"`）对所有请求做 token 提取与校验，`/login` 路径放行。
- **REQ-AUTH-004**：token 解析失败、签名不正确、已过期 SHALL 返回 HTTP 401。
- **REQ-AUTH-005**：系统 SHALL 提供 `/auth` 下的接口供客户端获取/校验当前用户的权限编码集合。

### 4.2 特性：基于角色的访问控制（RBAC）

#### 4.2.1 描述与优先级
**优先级：高**。系统采用 5 表 RBAC 模型：`users` / `roles` / `permissions` / `user_role` / `role_permission`。

#### 4.2.2 触发与响应
- **触发**：用户访问受控接口；或管理员维护角色/权限/分配关系。
- **响应**：拥有权限放行；缺失权限返回 HTTP 403。

#### 4.2.3 功能需求
- **REQ-RBAC-001**：系统 SHALL 提供用户管理接口（`/user`）：增、删、改、查、按角色查询、按用户查询其角色等共约 6 个接口。
- **REQ-RBAC-002**：系统 SHALL 提供角色管理接口（`/role`）：CRUD 共约 5 个接口；写操作需要 `SYS_ROLE_MANAGE` 权限。
- **REQ-RBAC-003**：系统 SHALL 提供权限查询接口（`/permission`）：只读，约 3 个接口；GET 请求白名单放行，便于前端构建菜单。
- **REQ-RBAC-004**：系统 SHALL 提供用户-角色分配接口（`/user-role`）共约 4 个，需要相应权限。
- **REQ-RBAC-005**：系统 SHALL 提供角色-权限分配接口（`/role-permission`）共约 4 个，需要 `SYS_PERMISSION_ASSIGN` 权限。
- **REQ-RBAC-006**：权限编码 SHALL 采用 `模块_功能` 命名格式（如 `SYS_USER_MANAGE`）。
- **REQ-RBAC-007**：`PermissionInterceptor` SHALL 通过 `@RequirePermission` 注解判定接口权限：
  - 支持方法级与类级注解，方法级优先；
  - 支持 `Logical.AND`（全部满足）与 `Logical.OR`（任一满足）；
  - 校验失败返回 HTTP 403；
  - 排除路径：`/login`、`/error`、`/auth/**`；`GET /role/**`、`GET /permission/**` 白名单放行。

### 4.3 特性：生产计划（Plan）管理

#### 4.3.1 描述与优先级
**优先级：高**。生产计划是顶层生产任务。

#### 4.3.2 触发与响应
- **触发**：生产主管创建或调度计划。
- **响应**：CRUD 与状态动作（PUBLISH/CANCEL_PUBLISH/PAUSE/RESUME/TERMINATE）按需执行；状态变更联动子级订单。

#### 4.3.3 功能需求
- **REQ-PLAN-001**：系统 SHALL 提供 `/plan` 路径下的 CRUD 接口及分页查询；分页参数为 `page`、`pageSize`。
- **REQ-PLAN-002**：Plan 实体 SHALL 至少包含字段：`planNo`、`planName`、`bomId`、`planNum`、`completed`、`startTime`、`endTime`、`priority`、`status`、`creatorId`、`publisher` 等。
- **REQ-PLAN-003**：系统 SHALL 提供状态动作接口 `POST /plan/{planNo}/actions/{action}`，支持 PUBLISH、CANCEL_PUBLISH、PAUSE、RESUME、TERMINATE。
- **REQ-PLAN-004**：状态转换 SHALL 严格遵循 `planStateMachine` 的规则：
  - `CREATED → {PUBLISH}`
  - `RELEASED → {CANCEL_PUBLISH, TERMINATE}`
  - `RUNNING → {PAUSE}`
  - `PAUSED → {RESUME}`
  - `COMPLETED/TERMINATED → {}`（终态）
  非法转换 SHALL 返回 HTTP 400（`IllegalStateException`）。
- **REQ-PLAN-005**：仅生产主管角色可执行 PUBLISH/CANCEL_PUBLISH/PAUSE/RESUME/TERMINATE；权限不足 SHALL 返回 HTTP 403（`SecurityException`）。
- **REQ-PLAN-006**：PUBLISH 动作 SHALL 触发 GatePolicy 三级门禁（产能、工艺、技能），任一失败返回 HTTP 400（`BusinessException`）。
- **REQ-PLAN-007**：PUBLISH 后，系统 SHALL 根据 BOM 对应的工艺流程与可用产线，自动拆分生成多个 Order（CREATED 状态）。

### 4.4 特性：生产订单（Order）管理

#### 4.4.1 描述与优先级
**优先级：高**。生产订单是 Plan 到具体产线的分解。

#### 4.4.2 功能需求
- **REQ-ORD-001**：系统 SHALL 提供 `/order` 下约 9 个接口，覆盖 CRUD、状态动作与多维度查询（按 Plan、按产线、按状态等）。
- **REQ-ORD-002**：Order 实体 SHALL 至少包含：`orderNo`、`planNo`（FK）、`lineNo`（FK）、`orderName`、`quantity`、`quantityProduced`、`qualifiedProducts`、`defectiveProducts`、`status`、`startTime/endTime`、`actualStart/End` 等。
- **REQ-ORD-003**：状态动作支持 PUBLISH/CANCEL_PUBLISH/PAUSE/RESUME/START_WORK/FINISH_WORK/TERMINATE，遵循 `orderStateMachine`：
  - `CREATED → {PUBLISH}`
  - `RELEASED → {CANCEL_PUBLISH, START_WORK, TERMINATE}`
  - `RUNNING → {PAUSE, FINISH_WORK}`
  - `PAUSED → {RESUME}`
  - 终态：COMPLETED、TERMINATED。
- **REQ-ORD-004**：仅车间主任、生产主管可执行 PAUSE/RESUME/TERMINATE；START_WORK/FINISH_WORK 由具备相应权限的用户触发。
- **REQ-ORD-005**：PUBLISH Order 时，系统 SHALL 根据工艺流程的工序与技能矩阵，为每个工序-员工组合自动派生 WorkOrder，并自动进入 RELEASED 状态。
- **REQ-ORD-006**：START_WORK 动作 SHALL 锚定 `actualStartTime`；FINISH_WORK 动作 SHALL 锚定 `actualEndTime`。
- **REQ-ORD-007**：TERMINATE 前 SHALL 校验：不存在 RUNNING 子工单；否则返回 HTTP 400。

### 4.5 特性：工单（WorkOrder）管理

#### 4.5.1 描述与优先级
**优先级：高**。工单是最细粒度的执行单元。

#### 4.5.2 功能需求
- **REQ-WO-001**：系统 SHALL 提供 `/workOrder` 下约 9 个接口，覆盖 CRUD、状态动作及多维度查询（按 Order、按工序、按员工、按状态等）。
- **REQ-WO-002**：WorkOrder 实体 SHALL 至少包含：`workOrderNo`、`orderNo`（FK）、`processId`、`userId`、`isCritical`、`plannedQty`、`actualQty`、`scrapQty`、`status`、`startTime/endTime`、`actualStart/End` 等。
- **REQ-WO-003**：状态动作支持 PUBLISH/CANCEL_PUBLISH/PAUSE/RESUME/START_WORK/FINISH_WORK/TERMINATE，遵循 `workOrderStateMachine`：
  - `CREATED → {PUBLISH}`
  - `RELEASED → {CANCEL_PUBLISH, START_WORK, TERMINATE}`
  - `RUNNING → {PAUSE, FINISH_WORK}`
  - `PAUSED → {RESUME, TERMINATE}`
  - 终态：COMPLETED、TERMINATED。
- **REQ-WO-004**：员工角色可执行 START_WORK/FINISH_WORK；车间主任/生产主管可 TERMINATE。
- **REQ-WO-005**：START_WORK 前 SHALL 校验该工单的 `userId` 是否具备对应工序的技能（与技能矩阵一致）；不一致返回 HTTP 400。
- **REQ-WO-006**：仅 `isCritical = true` 的关键工单的状态变更 SHALL 触发向上聚合到 Order；非关键工单仅记录自身状态。

### 4.6 特性：三层级联状态联动

#### 4.6.1 描述与优先级
**优先级：高**。Plan → Order → WorkOrder 状态变更通过级联在层级间传播，是系统的核心架构特征。

#### 4.6.2 向下联动需求
- **REQ-CASC-D-001**：`Plan PUBLISH` SHALL 根据工艺流程和可用产线，拆分为多个 Order（CREATED 状态）。
- **REQ-CASC-D-002**：`Plan PAUSE` SHALL 级联暂停所有 RUNNING 状态的 Order，并进一步级联暂停其下 RUNNING 的 WorkOrder。
- **REQ-CASC-D-003**：`Plan RESUME` SHALL 级联恢复所有 PAUSED 状态的 Order 与其下 PAUSED 的 WorkOrder。
- **REQ-CASC-D-004**：`Plan CANCEL_PUBLISH` 或 `Plan TERMINATE` SHALL 级联作废所有非终态的 Order 与其下非终态的 WorkOrder。
- **REQ-CASC-D-005**：`Order PUBLISH` SHALL 为产线的各工序生成 WorkOrder 并自动 RELEASED。
- **REQ-CASC-D-006**：`Order PAUSE/RESUME/TERMINATE` SHALL 级联到所有关联 WorkOrder。

#### 4.6.3 向上同步需求
- **REQ-CASC-U-001**：WorkOrder 状态变更（仅 `is_critical = true`）SHALL 触发向上聚合 Order 的状态：
  - 任一关键工单 RUNNING → Order 置为 RUNNING；
  - 全部 COMPLETED → Order 置为 COMPLETED；
  - 任一 PAUSED → Order 置为 PAUSED。
- **REQ-CASC-U-002**：Order 状态变更 SHALL 触发 `syncPlanStatusFromOrders()` 单次遍历所有 Order（O(N)），按 RUNNING > COMPLETED > PAUSED 优先级聚合 Plan 状态。

#### 4.6.4 状态服务标准管道（8 步）
- **REQ-PIPE-001**：所有状态服务（`planStateServiceImpl` / `orderStateServiceImpl` / `workOrderStateServiceImpl`）的 `handle()` 方法 SHALL 遵循统一的 8 步流程：
  1. 加载业务对象，构建 `StateContext{bizNo, action, userId}`；
  2. 权限校验（`PermissionPolicy.check()`）；
  3. 状态机校验（`StateMachine.check()`）；
  4. 门禁校验（`GatePolicy.check()`，仅 PUBLISH 触发）；
  5. 业务校验（如 TERMINATE 前检查子对象、START_WORK 前检查员工技能）；
  6. 状态持久化（`fromStatus.next(action)` 计算新状态，按动作差异化锚定时间）；
  7. 写入审计（`AuditService` 写入 `status_history`）；
  8. 级联联动（`handleLinkage`）。

#### 4.7.1 描述与优先级

#### 4.7.2 功能需求

### 4.15 特性：AI 智能交互与自然语言操作

#### 4.15.1 描述与优先级
**优先级：中高**。通过接入大语言模型（LLM），用户在 AI 交互窗口中输入自然语言指令，AI 自动分析意图、识别所需操作、校验权限，并调用对应的系统接口完成数据查询或写入。本特性旨在降低 MES 系统的操作门槛，使非技术用户也能通过自然语言高效完成复杂的车间数据操作。

典型场景示例：
- "帮我把设备功能'全自动超声波清洗'添加到设备'清洗机-CL01'上"
- "查询产线 L-03 上所有正在运行的订单"
- "创建一张物料名称为'铝合金外壳'的 BOM，图号为 DWG-2024-001"
- "将计划 PLN-20240612-001 的状态改为发布"

#### 4.15.2 触发与响应
- **触发**：用户在 AI 对话窗口发送自然语言消息；系统将其与历史会话上下文一同提交给 LLM 进行分析。
- **响应**：LLM 返回文本回复（解释/确认/追问）或触发工具调用指令；工具调用在后端沙箱中执行，结果返回给 LLM 做最终总结后呈现给用户。

#### 4.15.3 功能需求

##### 4.15.3.1 会话管理

- **REQ-AI-001**：系统 SHALL 提供 `POST /ai/chat/sessions` 接口，创建新的 AI 对话会话，返回会话 ID。会话 SHALL 关联创建用户的 `userId`。
- **REQ-AI-002**：系统 SHALL 提供 `GET /ai/chat/sessions` 接口，返回当前用户的历史会话列表（分页），按更新时间倒序。
- **REQ-AI-003**：系统 SHALL 提供 `GET /ai/chat/sessions/{sessionId}` 接口，返回会话详情及完整消息历史。
- **REQ-AI-004**：系统 SHALL 提供 `DELETE /ai/chat/sessions/{sessionId}` 接口，允许用户删除其自有会话（级联删除所有关联消息）。
- **REQ-AI-005**：会话实体 SHALL 包含：`id`、`userId`、`title`（由首条消息摘要自动生成）、`status`（ACTIVE/ARCHIVED）、`createdTime`、`updatedTime`。
- **REQ-AI-006**：系统 SHOULD 自动将会话标题设为首条用户消息的前 50 个字符（截断时保留语义完整性）；用户 MAY 手动修改标题。
- **REQ-AI-007**：会话历史 SHALL 至少保留 90 天；超期会话 MAY 被自动归档或清理。归档策略 MAY 在配置文件中调整。
- **REQ-AI-007a**：系统 SHALL 在每次 AI 对话处理过程中自动持久化会话详情，包括会话元数据（标题、活跃时间、状态）的实时更新，无需用户显式调用"保存"接口。持久化与查询接口（REQ-AI-002 / REQ-AI-003）SHALL 共用同一数据存储，保证"写入即可查"。

##### 4.15.3.2 消息与对话

- **REQ-AI-008**：系统 SHALL 提供 `POST /ai/chat/sessions/{sessionId}/messages` 接口，向指定会话发送一条自然语言消息并触发 AI 处理。请求体 SHALL 包含 `content`（消息文本，必填）。
- **REQ-AI-009**：响应 SHALL 为 SSE（Server-Sent Events）流式推送，包含以下事件类型：
  - `thinking`：AI 正在思考（可选的前端"思考中"状态提示）；
  - `text_delta`：流式文本回复片段（LLM 自然语言响应）；
  - `tool_call`：AI 决定调用系统工具，包含工具名与参数（用于前端展示"正在执行 XXX 操作"）；
  - `tool_result`：工具执行结果摘要；
  - `done`：本轮处理完成，携带完整消息 ID 与 token 消耗统计；
  - `error`：处理过程中发生错误，携带错误信息。
- **REQ-AI-010**：消息实体 SHALL 包含：`id`、`sessionId`、`role`（user/assistant/system/tool）、`content`（文本 JSON 块，存储 Markdown 格式）、`toolCalls`（AI 请求的工具调用列表，JSON 格式）、`tokenUsage`（本轮消耗的 prompt/completion token 数）、`createdTime`。
- **REQ-AI-011**：系统 SHALL 在每次 AI 交互时，将最近 N 轮对话历史（默认 20 轮，MAY 配置）作为上下文传递给 LLM，保证多轮对话的连贯性。
- **REQ-AI-012**：系统 SHOULD 在上下文窗口中注入当前用户的简要信息（角色、姓名），便于 LLM 据此调整回答风格与权限说明。
- **REQ-AI-012a**：系统 SHALL 在每轮消息处理中以原子方式持久化以下对话详情：
  1. 用户原始自然语言输入（`role=user`）；
  2. AI 最终文本回复（`role=assistant`，Markdown 格式）；
  3. 本轮所有工具调用记录（`tool_calls` 字段，JSON 数组，含工具名、入参、执行结果摘要）；
  4. 本轮 token 消耗统计（`token_usage` 字段，含 prompt/completion/total）；
  5. 自动更新所属会话的 `updated_at`。

  以上写入 SHALL 在 `POST /ai/chat/sessions/{sessionId}/messages` 接口的处理流程内同步完成，作为该接口固有副作用，不另行暴露独立的"保存"REST 端点。

##### 4.15.3.3 意图识别与工具调用（Function Calling）

- **REQ-AI-013**：系统 SHALL 向 LLM 注册一组 MES 工具定义（Tool Definitions），使 LLM 能够根据用户意图选择并调用合适的工具。工具定义 SHALL 以 JSON Schema 格式描述每个工具的名称、描述、参数及其类型约束。
- **REQ-AI-014**：首期 SHALL 注册的工具覆盖以下 MES 操作类别（总工具数不少于 15 个）：

| 类别 | 工具名称 | 对应系统操作 | 操作类型 |
|------|----------|-------------|----------|
| 设备管理 | `search_equipment` | 按名称/型号/类型模糊搜索设备 | 查询 |
| 设备管理 | `get_equipment_detail` | 获取设备详情及其已关联的功能列表 | 查询 |
| 设备管理 | `add_equipment_function` | 为设备新增一条功能描述 | 写入 |
| 设备管理 | `update_equipment_function` | 更新设备功能描述 | 写入 |
| 设备管理 | `delete_equipment_function` | 删除设备功能描述 | 写入 |
| BOM 管理 | `search_bom` | 按图号/名称/材质搜索 BOM | 查询 |
| BOM 管理 | `create_bom` | 创建新的 BOM 记录 | 写入 |
| 工序管理 | `search_process` | 按名称搜索工序 | 查询 |
| 工步管理 | `search_work_step` | 按名称/关联设备搜索工步 | 查询 |
| 工步管理 | `add_work_step` | 创建新的工步记录 | 写入 |
| 产线管理 | `search_line` | 搜索产线信息 | 查询 |
| 生产管理 | `query_plan_status` | 查询生产计划状态 | 查询 |
| 生产管理 | `query_order_status` | 按产线/计划查询订单状态 | 查询 |
| 生产管理 | `query_work_order_status` | 按订单/员工查询工单状态 | 查询 |
| 班组管理 | `search_team` | 搜索班组信息 | 查询 |

> 以上为最小可行集合，后续 SHALL 根据使用反馈持续扩展，逐步覆盖工艺流程、BOM 修改、状态变更等更多操作。新增工具 SHALL 通过配置或代码注册方式完成，不应修改 AI Agent 核心调度逻辑。

- **REQ-AI-015**：工具定义 SHALL 支持 JSON Schema `parameters` 字段，明确标注每个参数的 `type`、`description`、`required` 以及当参数为枚举类型时的 `enum` 约束（如状态值 `CREATED/RELEASED/RUNNING/PAUSED/COMPLETED/TERMINATED`）。
- **REQ-AI-016**：当 LLM 返回工具调用指令时，系统 SHALL 在执行前校验参数完整性：必填参数缺失时，返回追问信息给 LLM 让其向用户补充询问；参数类型/枚举不合法时，返回校验错误。
- **REQ-AI-017**：系统 SHALL 支持单轮对话中 LLM 连续调用多个工具（最多 5 次串行工具调用），并将前序工具的结果作为后续决策的上下文。

##### 4.15.3.4 安全沙箱与权限控制

- **REQ-AI-018**：所有 AI 触发的系统操作 SHALL 在执行前经过与普通 HTTP 请求相同的权限校验链路，包括：
  1. JWT 认证（从 AI 请求的 `Authorization` 头提取）；
  2. RBAC 权限校验（匹配 `@RequirePermission` 注解所要求的权限编码，依据用户在数据库中的角色-权限绑定关系判定）；
  3. 状态机校验（如涉及状态变更，校验转换合法性）；
  4. 业务门禁校验（如涉及 PUBLISH 动作，依次校验产能/工艺/技能门禁）。
- **REQ-AI-019**：任何权限不足的操作 SHALL 被拒绝，AI 代理 SHALL 将拒绝原因（如"您没有设备管理权限"）通过 LLM 自然语言化后告知用户，而非直接返回 HTTP 403。
- **REQ-AI-020**：破坏性操作（DELETE、TERMINATE、CANCEL_PUBLISH）在执行前 SHALL 要求用户二次确认。LLM SHALL 在回复中明确指出即将执行的操作及可能的影响，并要求用户回复"确认"或"是"后方可执行。
- **REQ-AI-021**：系统 SHALL 对单次 AI 会话的工具调用做速率限制：同一会话每分钟最多 10 次工具调用；超过限制返回友好提示并建议稍后重试。
- **REQ-AI-022**：每条 AI 消息的最大输入长度 SHALL 限制为 4000 字符；单次 AI 响应的最大 token 数 SHALL 可配置（默认 4096）。

##### 4.15.3.5 审计与可追溯性

- **REQ-AI-023**：每次 AI 触发的数据库写操作 SHALL 在 `status_history` 或专门的 AI 审计表中记录 `operatorId`（JWT 中的 userId）、`operationType`（CREATE/UPDATE/DELETE）、`targetType`（表名或实体名）、`targetId`、`aiSessionId`、`aiMessageId`、`requestPayload`（用户原始自然语言输入）和 `toolPayload`（AI 选择的工具与参数 JSON）。
- **REQ-AI-024**：AI 会话中所有的工具调用（含失败调用）SHALL 在 `ai_audit_log` 表中持久化记录，便于事后审计与问题排查。
- **REQ-AI-025**：系统 SHOULD 提供 `/ai/audit` 查询接口，支持按时间范围、用户、操作类型筛选 AI 操作审计日志。

##### 4.15.3.6 LLM 配置与管理

- **REQ-AI-026**：LLM 连接配置 SHALL 通过 `application.yml` 外部化，包括：
  - `provider`：LLM 提供商（如 `claude`、`openai`、`custom`）；
  - `api-key`：API 密钥（MUST 以加密或环境变量方式注入，不得明文硬编码）；
  - `model`：模型名称（如 `claude-opus-4-8`、`gpt-4o`）；
  - `base-url`：API 地址（支持私有化部署端点）；
  - `timeout`：请求超时时间（默认 60 秒）；
  - `max-tokens`：单次响应最大 token 数（默认 4096）；
  - `temperature`：生成温度（默认 0.3，偏向确定性以提升操作准确性）。
- **REQ-AI-027**：系统 SHALL 通过 `AiProvider` 接口抽象 LLM 调用，支持通过配置切换提供商或模型，无需修改核心 Agent 代码。
- **REQ-AI-028**：系统 SHALL 在 LLM API 调用失败时执行自动重试（最多 3 次，指数退避），超时/网络错误返回友好提示；MAY 提供降级策略（如切换至备用模型）。
- **REQ-AI-029**：系统 SHALL 记录每次 LLM API 调用的延迟与 token 消耗，提供 `/ai/metrics` 接口供运维监控（含平均延迟、成功率、token 消耗趋势等）。

#### 4.15.4 数据模型设计要点

AI 交互功能引入以下新数据库表：

> 上述表格中 `ai_chat_session` 与 `ai_chat_message` 即为 REQ-AI-007a / REQ-AI-012a 所述会话详情持久化的物理载体；查询接口（REQ-AI-002 / REQ-AI-003）从这两张表读取数据。

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `ai_chat_session` | AI 对话会话 | `id`、`user_id`（FK→users）、`title`、`status`（ACTIVE/ARCHIVED）、`created_at`、`updated_at` |
| `ai_chat_message` | 对话消息（含用户消息、AI 回复、工具调用记录） | `id`、`session_id`（FK）、`role`（user/assistant/system/tool）、`content`（TEXT，存储 Markdown/JSON）、`tool_calls`（JSON）、`token_usage`（JSON）、`created_at` |
| `ai_audit_log` | AI 操作审计日志 | `id`、`session_id`、`message_id`、`user_id`、`tool_name`、`tool_params`（JSON）、`execution_result`（TEXT）、`target_type`、`target_id`、`success`、`error_message`、`duration_ms`、`created_at` |
| `ai_tool_registry` | 工具注册表（可选，如采用动态注册） | `id`、`tool_name`、`tool_label`、`category`、`function_class`（对应 Service 方法全限定名）、`schema_definition`（JSON Schema）、`requires_confirmation`、`enabled`、`created_at` |

#### 4.15.5 架构要点

- AI 交互模块（建议包路径 `com.xtax.ai`）SHALL 独立于现有业务模块，通过调用已有 Service 层接口实现操作，不直接访问 Mapper 层，也不绕过状态机/权限策略/门禁策略。
- AI Agent 调度器（`AiAgentService`）负责：接收消息 → 组装上下文 → 调用 LLM → 解析响应 → 执行工具调用沙箱 → 返回结果 → 写入审计。
- 工具执行器（`ToolExecutor`）负责：根据 LLM 返回的工具名与参数，反射/路由到对应的业务 Service 方法，执行前后做参数验证与权限校验。
- 与 `LoginFilter` 和 `PermissionInterceptor` 的关系：AI 接口的访问也需 JWT 验证（`/ai/**` 路径不可被 LoginFilter 放行）。工具调用阶段，`ToolExecutor` 应主动调用 `AuthService` 做权限检查，而非依赖拦截器（因工具调用发生在一次请求内部而非独立 HTTP 请求）。

### 5.2 安全需求

| 编号 | 要求 |
|------|------|
| **NFR-SEC-001** | 所有受控接口 SHALL 通过 JWT 认证 + RBAC 授权 + 业务策略 + 业务门禁 + 状态机的 5 层防线（详见概要设计 8.4 节）。 |
| **NFR-SEC-002** | JWT 密钥 MUST 通过应用配置管理；生产环境 MUST 使用足够强度的密钥（≥ 256 位）。 |
| **NFR-SEC-003** | JWT 有效期 SHALL 为 24 小时；过期后客户端 MUST 重新登录。 |
| **NFR-SEC-004** | 密码 SHALL 在数据库中以加盐哈希形式存储，禁止明文。 |
| **NFR-SEC-005** | 所有写操作 SHALL 记录 `operatorId`；状态变更 SHALL 写入 `status_history`。 |
| **NFR-SEC-006** | 生产环境 SHALL 启用 HTTPS，CORS 来源 SHOULD 收紧白名单（默认全放通仅适用于开发/测试）。 |
| **NFR-SEC-007** | 任何未捕获异常 SHALL 由 `globalExceptionHandler` 兜底返回 500，避免堆栈泄漏。 |
| **NFR-SEC-008** | AI 接口（`/ai/**`）SHALL 受 JWT 认证保护，不可由 `LoginFilter` 放行；AI 请求与普通 HTTP 请求使用相同的认证链路。 |
| **NFR-SEC-009** | LLM API Key MUST 通过环境变量或加密配置注入，禁止以明文存储在 `application.yml` 或代码仓库中。 |
| **NFR-SEC-010** | 用户通过 AI 对话提交的自然语言内容 SHALL 遵循输入长度限制（≤ 4000 字符），防止 prompt 注入攻击耗尽 token 预算。 |

| **NFR-SEC-012** | 敏感操作（删除、终止、取消发布）当由 AI 触发时 SHALL 要求用户显式确认；确认提示中 MUST 说明操作的目标对象与不可逆后果。 |
